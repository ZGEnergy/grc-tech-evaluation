# G-FNM-3: DCPF verification against reference solution on cleaned FNM

## Result: PASS

## Approach

Loaded the pre-cleaned FNM main island network from the MATPOWER fallback
path (`/workspace/data/fnm/reference/cleaned/fnm_main_island.mat`) via
`loadcase()`. Solved DCPF using `rundcpf()` with default solver settings
(verbose off, output off). Compared bus voltage angles and branch active
power flows against the reference DCPF solution in
`data/fnm/reference/dcpf/buses_dcpf.csv` and `branches_dcpf.csv`.

The reference solution was itself generated by MATPOWER from the same
`.mat` file, so zero deviation is the expected outcome. This test confirms
that the MATPOWER fallback ingestion path preserves the network data
exactly and that `rundcpf()` produces a deterministic solution.

The v11 bus injection power balance check was also performed, verifying
that at every non-excluded bus, the net injection (generation minus load
minus shunt) equals the net branch flow leaving the bus.

Excluded 2,445 buses per `excluded_buses.json` (IDE=4 isolated buses and
disconnected islands).

### Interpretation

All deviations are at float64 machine noise level (~5e-9 degrees for bus
angles, ~5e-7% for branch flows). This is expected because the reference
DCPF solution was generated by the same MATPOWER version from the same
`.mat` file. The test confirms that:

1. The MATPOWER fallback ingestion path (`loadcase()` on `.mat` file)
   preserves the cleaned FNM network without any data loss or transformation
2. `rundcpf()` produces a deterministic DCPF solution
3. Bus injection power balance is satisfied to machine precision (max
   mismatch 6.2e-9 MW)

For cross-tool comparison, this result establishes MATPOWER as the
reference implementation. Other tools' deviations from the DCPF reference
are attributable to formulation differences (e.g., B-matrix construction
with/without tap ratios) or data ingestion fidelity, not to MATPOWER
solver error.
