# C-4: SCUC Scale — SMALL (ACTIVSg 2000-bus)

## Result: QUALIFIED PASS

## Approach

Scaled the A-5 TINY user-assembled JuMP MILP approach to the 2000-bus ACTIVSg network. PowerModels v0.21.5 does NOT natively support SCUC; the entire UC formulation (~300 LOC) was user-assembled using JuMP with PowerModels serving only as a data parser (`parse_file`).

### Implementation Details

1. **Data loading:** `PowerModels.parse_file("case_ACTIVSg2000.m")` — used native MATPOWER cost models.
2. **UC parameters:** Generators classified by capacity as proxy for technology type (peaker < 50 MW, mid 50-200 MW, baseload 200-500 MW, nuclear > 500 MW). Min up/down times and ramp rates assigned per class.
3. **Load profile:** 24-hour profile generated by scaling base load with typical summer day multipliers (0.68x trough at H4, 1.0x peak at H12/H18).
4. **Formulation:** Same as A-5 TINY: commitment bounds, startup/shutdown logic, min up/down time, ramp rates, nodal power balance with B-theta DC power flow, branch thermal limits.
5. **Solvers tested:** HiGHS (1 thread, 30-min timeout), SCIP (1 thread, 30-min timeout).

## Workarounds

- **What:** Entire SCUC formulation is user-assembled as a JuMP MILP (~300 LOC of model construction code). PowerModels provides only data parsing.
- **Why:** PowerModels v0.21.5 is a steady-state single-period network optimization library. SCUC (binary commitment variables, min up/down time, startup/shutdown costs, multi-period coupling) falls entirely outside its scope.
- **Durability:** blocking -- No API path exists (public or private) to achieve SCUC. The workaround requires writing a substantial custom optimization model from scratch.
- **Grade impact:** C or below for the SCUC sub-question. At SMALL scale, the user-assembly effort is even more significant: the ~300 LOC model produces 100K+ variables and 340K constraints. However, the JuMP foundation makes this feasible, and SCIP successfully solves it.
