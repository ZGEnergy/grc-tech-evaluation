# A-3: Solve DC OPF with Generator Costs and Line Flow Limits

## Result: QUALIFIED PASS

## Approach

Loaded IEEE 39-bus network via the standard import path. Since PyPSA's pypower importer
**does not import gencost data** from MATPOWER files, all `marginal_cost` values default
to 0.0. This required a manual workaround to assign generator costs.

The MATPOWER case39 uses identical quadratic cost functions for all generators:
`cost = 0.01*p^2 + 0.3*p + 0.2`. Since PyPSA's DC OPF is a linear program, we linearized
the quadratic cost around each generator's initial operating point:
`marginal_cost = c1 + 2*c2*p_set`.

Solved DC OPF using `net.optimize(solver_name="highs")` with HiGHS solver settings per
the evaluation solver configuration (time_limit=300, presolve=on, threads=1).

Line flow limits (`s_nom`) were already set from the MATPOWER `rateA` values by the
pypower importer. All 35 lines and 11 transformers had positive flow limits.

## Workarounds

- **What:** Manually assigned `marginal_cost` on each generator from MATPOWER `gencost` data,
  linearizing the quadratic cost function around the initial operating point.
- **Why:** PyPSA's `import_from_pypower_ppc` explicitly skips `gencost` data. The import
  warning states: "some PYPOWER features not supported: areas, gencosts, component status".
- **Durability:** stable -- The `generators.marginal_cost` attribute is a documented public
  DataFrame column. Setting it via `net.generators.at[name, "marginal_cost"] = value` is
  standard PyPSA API usage. The workaround is in the data loading, not the solve.
- **Grade impact:** Minor. The OPF solve itself works perfectly. The friction is in the
  import path, not the optimization API. Any production workflow would have its own cost
  data source rather than relying on MATPOWER gencost import.
