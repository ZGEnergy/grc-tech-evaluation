# A-8: Multi-Period Stochastic DC OPF with Scenario-Indexed Timeseries

## Result: QUALIFIED PASS

## Approach

PyPSA 1.1.x has **native stochastic scenario support** via `n.set_scenarios()`. This
creates a multi-scenario optimization where:

- Components (generators, loads, lines) are internally replicated per scenario
- The objective is probability-weighted: E[cost] = sum(weight_s * cost_s)
- All scenarios are co-optimized in a **single LP** (not separate deterministic solves)
- Scenario-specific time-varying data uses MultiIndex columns (scenario, component)

**Test setup:**
- 3 scenarios: low (25%), mid (50%), high (25%)
- 24-hour snapshots with diurnal load profile
- Load varied by +/-5% across scenarios (correlated perturbation)
- Wind generator (500 MW on bus 25) with scenario-dependent capacity factors
- Wind CF anti-correlated with load (higher wind in low-demand scenarios)

**Required workaround:** PyPSA 1.1.2 has a bug in `SubNetwork.find_bus_controls()`
when used with scenarios on pypower-imported networks. The method attempts to look up
buses by non-scenario index in a MultiIndex, causing a KeyError. Monkey-patching
`find_bus_controls` to no-op is safe for DC OPF (PV/PQ classification only affects
AC power flow).

genuine and not just replicated deterministic solves.

## Workarounds

1. **What:** Manually assigned `marginal_cost` from MATPOWER gencost data.
   - **Why:** PyPSA pypower importer skips gencost on import.
   - **Durability:** stable -- uses documented public API.
   - **Grade impact:** Minor, same as A-3.

2. **What:** Monkey-patched `SubNetwork.find_bus_controls` to no-op.
   - **Why:** PyPSA 1.1.2 bug -- `find_bus_controls()` uses non-scenario bus index
     to look up PV generators after `set_scenarios()` creates MultiIndex. This causes
     `KeyError: "Index(['30', '32', ...] not in index"`.
   - **Durability:** fragile -- relies on monkey-patching an internal method. The fix
     is safe for DC OPF but would break AC PF scenarios. This is likely a known or
     soon-to-be-fixed bug in PyPSA given that scenarios are a new feature.
   - **Grade impact:** The stochastic API itself (`set_scenarios`, `get_scenario`,
     scenario-indexed DataFrames) is well-designed and genuinely native. The bug is in
     the interaction between the legacy pypower importer and the new scenario system.
     A network built natively in PyPSA (not via pypower import) would likely avoid
     this issue entirely.

Key API calls:

```python
# Native scenario setup
net.set_scenarios({"low": 0.25, "mid": 0.50, "high": 0.25})

# Set base time-varying data before set_scenarios() -- auto-duplicated
net.loads_t.p_set = load_df_base

# Differentiate per scenario (MultiIndex columns)
for scenario in ["low", "mid", "high"]:
    net.loads_t.p_set[(scenario, load_name)] *= LOAD_MULTIPLIERS[scenario]
    net.generators_t.p_max_pu[(scenario, "Wind_25")] *= WIND_MULTIPLIERS[scenario]

# Single stochastic optimization
status, termination = net.optimize(solver_name="highs", solver_options={...})

# Extract per-scenario results
scen_gen_cols = [c for c in net.generators_t.p.columns if c[0] == scenario]
scen_dispatch = net.generators_t.p[scen_gen_cols]
```

The stochastic API is a first-class feature in PyPSA 1.1.x: `set_scenarios()` creates
a genuine multi-scenario formulation where the optimizer sees all scenarios simultaneously
with probability-weighted costs. The `get_scenario()` method (not used here due to a
deepcopy bug with HiGHS) extracts single-scenario results from the joint solution.
