# C-5: N-M Contingency Sweep at Scale (MEDIUM — 10,000-bus network)

## Result: FAIL (timeout — combinatorial explosion)

## Approach

Attempted N-M contingency sweep on the 10,000-bus ACTIVSg10k network with parameters
x=5 (graph distance from highest-degree bus) and m=4 (max simultaneous outages).

The test script:
1. Builds NetworkX graph from `net.graph()`
2. Finds all branches within graph distance 5 of the highest-degree bus
3. Enumerates combinations for N-1 through N-4 outages
4. Solves DCPF via `net.lpf()` per contingency case
5. Prunes branches causing load loss

## Output

**Not completed.** The test was terminated after exceeding the evaluation time budget
(multiple attempts, each >300s). The root cause is combinatorial explosion in the
parameter space:

On the 10k-bus network with x=5, the number of branches in scope is very large
(estimated 200+ branches within graph distance 5 of the highest-degree hub bus).
The N-4 combinations alone would produce C(200,4) ≈ 64 million cases, each requiring
a sparse LU factorization (~6s per DCPF on this network from C-1). Even with aggressive
pruning, this is computationally infeasible.

**Contrast with A-7 (TINY, x=3, m=3):** On the 39-bus network, x=3 yields ~15-20
branches, producing manageable combination counts (C(20,3) = 1,140 cases). The
parameters do not scale linearly with network size.

## Workarounds

None attempted. The issue is not a PyPSA limitation but a parameter scaling problem
in the test protocol. Possible mitigations:

- **Reduce parameters for MEDIUM:** x=2, m=2 would be more appropriate for 10k-bus
- **Use BODF-based screening:** PyPSA has `lpf_contingency()` (BODF method) but it's
  broken in v1.1.2 (DataFrame.to_frame bug, see B-3 observation)
- **Cap enumeration:** Add a max-cases threshold (e.g., 10,000) to prevent runaway

## Evaluation Process Note

This test exposed a protocol design issue: the contingency sweep parameters (x=5, m=4)
from the eval-config are specified for the MEDIUM grade network but were calibrated
based on TINY network behavior. The combinatorial growth of C(n, m) with network
density means these parameters need tier-specific scaling. This finding applies to
all tools, not just PyPSA.
