# F-6: Distribution integrity

## Finding
PyPSA is distributed via PyPI with semantic versioning, Trusted Publishing (Sigstore attestation), and conda-forge. The release pipeline is automated through GitHub Actions with provenance attestations.

## Evidence

### PyPI distribution
- **Package:** <https://pypi.org/project/pypsa/>
- **Latest version:** 1.1.2 (released 2026-02-23)
- **Total releases:** 85 versions, spanning from v0.3 (2016-01) to v1.1.2 (2026-02)
- **Trusted Publishing:** Yes -- releases are signed via Sigstore attestation bundles
- **Release workflow:** `release.yml` GitHub Action, triggered by git tags
- **Yanked versions:** v1.1.1 was yanked due to CI pipeline error, with users directed to v1.1.2

### Versioning scheme
Semantic versioning (MAJOR.MINOR.PATCH). The v0.x to v1.0 transition occurred in 2024-2025 timeframe, indicating maturity milestone.

### Conda-forge
Also available via `conda install -c conda-forge pypsa`, providing an independent distribution channel with its own build verification.

### Lock file integrity
The project's `uv.lock` file pins all dependencies with SHA-256 hashes of wheel/sdist files, enabling verification of downloaded artifacts against known-good hashes.

## Implications
Strong distribution integrity. Sigstore attestation provides cryptographic provenance linking PyPI artifacts to the GitHub Actions build. Two independent distribution channels (PyPI + conda-forge) provide supply chain redundancy. The yanking of v1.1.1 demonstrates responsible release management.
