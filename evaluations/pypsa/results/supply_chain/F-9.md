# F-9: Example artifact integrity

## Finding
PyPSA's built-in example networks are pinned to the exact release version of the installed package. The `_repo_url()` function constructs GitHub URLs using the version tag, ensuring examples always match the installed code.

## Evidence

### Version-pinned example URLs
From `pypsa/examples.py`:

```python
from pypsa.version import __version_base__

def _repo_url(master=False, url="https://github.com/PyPSA/PyPSA/raw/"):
    if master or parse_version(__version_base__) < parse_version("0.35.0"):
        return f"{url}master/"
    return f"{url}v{__version_base__}/"
```

For the installed version (1.1.2):

```
>>> _repo_url()
'https://github.com/PyPSA/PyPSA/raw/v1.1.2/'
```

This means `pypsa.examples.ac_dc_meshed()` fetches from:
`https://github.com/PyPSA/PyPSA/raw/v1.1.2/examples/networks/ac-dc-meshed/ac-dc-meshed.nc`

### Version pinning activated since v0.35.0
The code explicitly checks: versions >= 0.35.0 use the version tag; older versions fall back to `master`. This was a deliberate design decision to ensure example reproducibility.

### Master override available
A `master=True` parameter allows users to explicitly request the latest development examples, but the default is always the pinned release version.

### Exception: carbon_management example
The `carbon_management()` example fetches from an external URL (`tubcloud.tu-berlin.de`) that is not version-pinned. This is a single large example network hosted externally. If unavailable, it raises a `RuntimeError` with a clear message.

## Implications
Strong example integrity. Users get examples that match their installed version by default. The version-pinning mechanism prevents the common failure mode of examples breaking due to API changes between releases. The one exception (carbon_management) is clearly documented and fails gracefully.
