# D-4: Error quality

## Finding

PyPSA produces meaningful diagnostics for two of three deliberate error scenarios. The
infeasible OPF and missing cost curve cases surface clear error messages. The invalid
bus type case fails silently — the tool accepts the invalid value without warning and
produces results as if nothing is wrong.

## Evidence

Tests run in devcontainer via `evaluations/pypsa/tests/accessibility/d4_error_test.py`.

### (a) Infeasible OPF — all line limits set to 0

Classification: Meaningful diagnostic

Setting all line and transformer `s_nom` to 0 and running `net.optimize()` produces:

```
Model status: Infeasible
```

The solver (HiGHS) detects infeasibility during presolve and reports it clearly. PyPSA's
linopy layer surfaces this as:

```
WARNING:linopy.constants:Optimization potentially failed:
Status: warning
Termination condition: infeasible
Solution: 0 primals, 0 duals
Objective: nan
```

The `optimize()` method returns `('warning', 'infeasible')` as a status tuple. The user
receives an unambiguous signal that the problem is infeasible. PyPSA also provides a
dedicated "Tracing Infeasibilities" example in the documentation for debugging such cases.

**Weakness:** After infeasibility, `net.objective` is `None`, which can cause a
`TypeError` if the user attempts `float(net.objective)` without checking. A more robust
API would return 0.0 or raise a dedicated exception.

### (b) Missing generator cost curve — all marginal_cost = 0

Classification: Meaningful diagnostic

When no generator costs are assigned (all `marginal_cost` remain at the default 0.0),
calling `net.optimize()` raises:

```
ValueError: Objective function could not be created. Please make sure the components
have assigned costs.
```

This is an excellent error message. It is raised at model construction time (before
solver invocation), identifies the exact problem (missing costs), and provides actionable
guidance. The traceback points to `pypsa/optimization/optimize.py:define_objective`.

### (c) Invalid bus type — bus control set to arbitrary string

Classification: Silent failure

Setting a bus `control` attribute to `"InvalidType"` or `"BOGUS_TYPE_999"` produces
**no error, no warning, and no validation failure**. Both `net.pf()` (Newton-Raphson
power flow) and `net.optimize()` (OPF) complete successfully and return valid-looking
results:

- `net.pf()` converges in 4 iterations with error 3.3e-09
- `net.optimize()` returns `('ok', 'optimal')`

PyPSA uses the `control` field only during power flow subnetwork construction. An
invalid value means the bus is not treated as PQ, PV, or Slack — but no validation
enforces that the value must be one of these. The bus is effectively silently ignored
in the power flow control classification, defaulting to some implicit behavior.

This is a significant gap. A typo in bus type assignment (e.g., `"PQ "` with trailing
space, or `"pq"` lowercase) would produce silently incorrect results with no diagnostic.

## Implications

PyPSA's error handling is strong at the optimization layer (tests a and b) but weak at
the data model layer (test c). The optimization stack (linopy + HiGHS) provides clear
infeasibility and missing-data diagnostics. However, the network data model accepts
arbitrary strings for categorical fields without validation, creating a class of silent
errors that would be difficult to debug in production use. Adding enum validation on
component attributes like `control` would close this gap.
