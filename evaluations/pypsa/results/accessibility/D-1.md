# D-1: Install-to-first-solve

## Finding

PyPSA installs quickly via `uv sync` (under 30 seconds for dependency resolution and
install). The core API for running a DCPF once data is loaded is minimal (`net.lpf()`).
However, reaching a successful first solve on a standard MATPOWER `.m` test case required
two workarounds for broken or absent import paths, adding significant friction to the
install-to-first-solve experience.

**Total elapsed time from clean environment to first successful DCPF solve:** approximately
15-20 minutes of wall-clock effort, dominated by debugging import path failures, not by
installation itself.

## Evidence

From `evaluations/pypsa/notes/install-findings.md`:

1. **No native MATPOWER `.m` reader.** PyPSA cannot load `.m` files directly. The user
   must bring in `matpowercaseframes` (a third-party package) or `pandapower` as an
   intermediary.

2. **`import_from_pandapower_net()` crashes on case39.** A shape mismatch in
   `v_mag_pu_set` assignment (bus count vs generator count broadcast failure) when
   multiple generators share a bus (bus 31 in case39). This is a bug that ships in
   PyPSA 1.1.2.

3. **`import_from_pypower_ppc()` works but requires manual dict construction.** The user
   must construct a pypower-format dictionary from `matpowercaseframes` output, not a
   trivial step for a new user. It emits warnings about unsupported features (areas,
   gencosts, component status).

4. **Once loaded, the API is clean.** `net.lpf()` runs DCPF with no additional
   configuration. Results are in pandas DataFrames.

### Issues encountered (severity):

| Issue | Severity |
|-------|----------|
| No `.m` file reader — must use third-party intermediary | Medium |
| Pandapower import path crashes on standard test case | High |
| Pypower import path requires manual dict construction | Medium |
| Pypower import silently drops gencost data | Medium |
| Warnings about unsupported features are informational only | Low |

## Implications

A new user starting from a MATPOWER `.m` file will not achieve a solve on the first
attempt. The most obvious import path (`import_from_pandapower_net`) crashes, and the
working path (`import_from_pypower_ppc`) requires knowledge of pypower's internal data
format. This is a meaningful barrier for power-systems engineers accustomed to MATPOWER's
`runpf('case39')` one-liner. The barrier is entirely in data ingestion, not in the
solve API itself.
