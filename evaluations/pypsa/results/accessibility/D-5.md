# D-5: Code volume comparison

## Finding

PyPSA Suite A test scripts range from 86 to 142 SLOC (source lines of code, excluding
blank lines, comments, and docstrings). Approximately 30-40% of each script is shared
boilerplate (imports, MATPOWER loading helper, result recording structure). The core
problem-specific code for each test is relatively compact.

Only tests A-1 through A-3 have been implemented at the time of this evaluation. Tests
A-4 through A-8 have not yet been written, so this is a partial measurement.

## Evidence

### LOC counts

| Script | Total lines | Non-blank | SLOC | Imports | Core solve logic (est.) |
|--------|-------------|-----------|------|---------|------------------------|
| `test_a1.py` (DCPF) | 127 | 102 | 86 | 7 | ~20 |
| `test_a2.py` (ACPF) | 199 | 165 | 141 | 7 | ~50 |
| `test_a3.py` (DC OPF) | 208 | 174 | 142 | 7 | ~55 |

Files located at: `evaluations/pypsa/tests/expressiveness/test_a{1,2,3}.py`

### Shared boilerplate across all three scripts

Each script contains approximately 35-40 lines of shared structure:
- Imports (7 lines): `pypsa`, `matpowercaseframes`, `json`, `time`, `traceback`, `Path`
- `_load_network()` helper (10 lines): MATPOWER-to-pypower dict construction
- `run()` result dict initialization (8 lines): status, wall_clock, details, errors
- `__main__` block (3 lines): execute and print JSON
- Exception handling wrapper (5-8 lines)

### Core problem-specific code estimates

- **A-1 (DCPF):** ~20 lines — load network, call `net.lpf()`, extract bus angles and
  line flows from DataFrames, validate non-zero results.
- **A-2 (ACPF):** ~50 lines — load network, call `net.pf()`, check convergence info,
  implement DC warm-start fallback, extract V magnitude/angle and P/Q flows, compute
  losses.
- **A-3 (DC OPF):** ~55 lines — load network, manually assign gencost from
  MATPOWER data (workaround for missing import), call `net.optimize()`, extract
  dispatch, LMPs, and shadow prices.

### Observations

1. The MATPOWER loading workaround (`_load_network`) adds ~10 lines to every script
   that would be unnecessary if PyPSA had a native `.m` file reader.
2. The A-3 gencost workaround adds ~15 lines that would be unnecessary if the pypower
   importer preserved cost data.
3. PyPSA's core API calls are compact: `net.lpf()` (1 line), `net.pf()` (1 line),
   `net.optimize()` (1-3 lines with solver options).
4. Result extraction is verbose because the test protocol requires explicit validation
   of output structure (DataFrame shapes, non-zero checks).

## Implications

PyPSA's API is concise for the core solve operations. The code volume is inflated by
data ingestion workarounds (MATPOWER import limitations) and by the test harness
structure (result recording, validation assertions). For a fair cross-tool comparison,
the shared boilerplate and test harness should be excluded, focusing on the
problem-specific code. The core solve logic for DCPF is approximately 20 lines, which
is competitive with other tools in this evaluation. Full cross-tool comparison will be
possible once all tools have completed Suite A implementation.
