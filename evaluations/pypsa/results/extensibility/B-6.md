# B-6: Code architecture (qualitative)

## Result: PASS

## Approach

Traced the DCPF solve path from the public API call `Network.lpf()` through to the
final `scipy.sparse.linalg.spsolve` invocation. Examined source code in PyPSA 1.1.2,
specifically `pypsa/network/power_flow.py` (~1860 lines) and `pypsa/network/graph.py`
(~300 lines).

## Finding

PyPSA's DCPF architecture has **clear separation of concerns across 4 abstraction layers**
with good internal documentation, but has some coupling issues and two regression bugs
in v1.1.2 that indicate insufficient test coverage for edge cases.

## Evidence

### Abstraction Layers (API call to solver)

**Layer 1: Network-level dispatch** (`NetworkPowerFlowMixin.lpf`, line 805)
- Entry point: `net.lpf(snapshots, skip_pre)`
- Delegates to `_network_prepare_and_run_pf(n, sns, skip_pre, linear=True)`
- Responsibility: snapshot normalization, choosing linear vs. nonlinear path
- LOC: ~5 (thin dispatcher)

**Layer 2: Network preparation and sub-network iteration** (`_network_prepare_and_run_pf`, line 143)
- Determines network topology (connected components)
- Allocates output DataFrames
- Iterates over sub-networks, calling each sub-network's `lpf()`
- Handles Link components (controllable branches)
- Selects preparation function (`calculate_B_H` for linear, `calculate_Y` for nonlinear)
- LOC: ~90

**Layer 3: Sub-network LPF solver** (`SubNetworkPowerFlowMixin.lpf`, line 1746)
- Operates on a single connected sub-network
- Computes nodal power injections from all one-port and controllable branch components
- Calls `calculate_B_H()` to build susceptance matrices
- Solves `B * theta = P` via `scipy.sparse.linalg.spsolve`
- Computes branch flows as `H * theta`
- Assigns results back to dynamic DataFrames
- Handles slack bus power balancing
- LOC: ~120

**Layer 4: Matrix construction** (`SubNetworkPowerFlowMixin.calculate_B_H`, line 1079)
- Builds incidence matrix K from branch bus0/bus1 topology
- Computes susceptance vector b = 1/x for each branch
- Forms `H = diag(b) * K^T` (branch-bus transfer matrix)
- Forms `B = K * H` (weighted Laplacian / susceptance matrix)
- Handles phase-shift transformers
- LOC: ~60

**Solver invocation** (line 1831):

```python
v_diff[:, 1:] = spsolve(self.B[1:, 1:], p[:, 1:].T).T
flows = pd.DataFrame(v_diff * self.H.T, columns=branches_i, index=sns) + self.p_branch_shift
```

### Separation of Concerns

| Concern | Location | Clean? |
|---------|----------|--------|
| Topology detection | `Network.determine_network_topology()` | Yes — separate method |
| Graph export | `NetworkGraphMixin.graph()` → NetworkX | Yes — clean mixin |
| Matrix assembly | `SubNetwork.calculate_B_H()` | Yes — isolated |
| PTDF/BODF | `SubNetwork.calculate_PTDF()`, `calculate_BODF()` | Yes — layered on B/H |
| Linear solve | `SubNetwork.lpf()` | Mostly — mixes injection aggregation with solve |
| Output allocation | `_allocate_pf_outputs()` | Yes — separate function |
| Contingency analysis | `Network.lpf_contingency()` | Yes — separate method using BODF |

### Architecture Pattern

PyPSA uses **mixin classes** to compose the Network class:
- `NetworkGraphMixin` — graph export, adjacency/incidence matrices
- `NetworkPowerFlowMixin` — power flow methods (lpf, pf, lpf_contingency)
- `_NetworkABC` — abstract base with type hints
- `SubNetworkPowerFlowMixin` — sub-network-level solvers

This is a reasonable pattern for a tool of this complexity. The mixins keep file sizes
manageable and allow independent evolution of graph, power flow, and optimization code.

### Internal Documentation

- All public methods have docstrings with parameter descriptions and return types
- `calculate_B_H` references "leans heavily on pypower.makeBdc" (provenance)
- `calculate_BODF` includes the mathematical formula: `f_k^after = f_k^before + BODF_{kl} * f_l^before`
- `calculate_PTDF` documents the mathematical construction step by step
- Module-level docstring: "Power flow functionality."
- Type hints are used extensively (both in mixins and via TYPE_CHECKING imports)

### Concerns

1. **Layer 3 (sub-network LPF) is too large:** The `SubNetwork.lpf()` method (~120 lines)
   mixes nodal injection aggregation, matrix solve, flow computation, and slack adjustment.
   These could be separate functions.

2. **Regression bugs in v1.1.2:** Two bugs in `power_flow.py`:
   - `lpf_contingency()` line 934: `p0_base.to_frame("base")` fails when `p0_base` is a
     DataFrame (happens with single snapshot)
   - `lpf()` line 1840: `flows.loc[:, c.name]` fails with KeyError when a sub-network has
     no branches of a given passive component type (happens when branch removal causes islanding)

   Both bugs suggest edge-case testing gaps. They are in public API methods that presumably
   worked in earlier versions.

3. **No intermediate result caching:** Each `lpf()` call recomputes topology, dependent values,
   and B/H matrices. For N-1 contingency loops using `copy()` + `lpf()`, this means ~46
   full matrix builds. The BODF approach (`lpf_contingency`) would avoid this but is broken.

### Strengths

- **Clean NetworkX integration:** `graph()` method returns a proper NetworkX MultiGraph with
  full edge provenance. No friction for graph algorithms.
- **Sparse linear algebra:** Uses `scipy.sparse` throughout — `csr_matrix`, `csc_matrix`,
  `spsolve`. Appropriate for power systems.
- **Consistent data model:** All results stored in `n.buses_t`, `n.lines_t`, etc. as
  pandas DataFrames indexed by snapshot. Uniform access pattern.
- **Copy support:** `net.copy()` provides clean deep clone for contingency analysis.
- **PTDF/BODF infrastructure:** The mathematical framework for contingency analysis exists
  even if the API wrapper has a bug.

## Implications

PyPSA's code architecture is well-structured for its scope. The 4-layer decomposition
(dispatch → topology iteration → sub-network solve → matrix construction) is clean and
traceable. The mixin pattern keeps concerns separated. Internal documentation is above
average for open-source power systems tools.

The two regression bugs in v1.1.2 are concerning but fixable — they are API-level issues,
not architectural deficiencies. The core solver path (Layer 3-4) is sound and uses
appropriate sparse linear algebra.

For extensibility specifically: the clean graph export, the copy mechanism, and the
separated matrix construction make it straightforward to build custom analysis on top
of PyPSA's internals without needing to understand the full codebase.
