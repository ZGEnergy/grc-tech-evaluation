# B-1: Add Flow Gate Limit to DC OPF

## Result: PASS

## Approach

Added a flow gate (interface) constraint to the DC OPF from A-3 using PyPSA's
`extra_functionality` callback. The callback provides access to the linopy optimization
model, allowing custom constraints to be added programmatically.

**Flow gate definition:** Lines L2, L3, L4 with positive direction coefficients.
The gate limit was set to 80% of the unconstrained baseline flow to force meaningful
re-dispatch.

The `extra_functionality` callback pattern:
1. Access the linopy model via `network.model`
2. Reference line flow variables via `m.variables["Line-s"]`
3. Select specific lines with `.sel(name=line_name)`
4. Build a linear expression and add constraints via `m.add_constraints()`

This uses only documented PyPSA/linopy APIs -- no source code modification or forking.

## Output

**Baseline (unconstrained DC OPF):**
- Objective: $76,890.14/hr
- Gate flow: 494.8 MW

**Constrained (with flow gate at 80%):**

The re-dispatch demonstrates that the gate constraint is binding and producing

replaced by more expensive generation on the other side.

**LMP range:** [5.30, 23.20] $/MWh (wider than baseline due to congestion)

## Workarounds

None required for the flow gate mechanism itself. The `extra_functionality` callback
is a documented, first-class extension point in PyPSA's optimization API.

(Marginal cost assignment workaround from A-3 applies to the underlying network setup.)

## API Mechanism

```python
def flow_gate_constraint(network, snapshots):
    m = network.model
    line_flow_var = m.variables["Line-s"]

    # Build gate flow expression
    gate_flow = sum(
        direction * line_flow_var.sel(name=line_name)
        for line_name, direction in zip(gate_lines, gate_directions)
    )

    # Add bidirectional limit
    m.add_constraints(gate_flow <= gate_limit, name="flow_gate_upper")
    m.add_constraints(gate_flow >= -gate_limit, name="flow_gate_lower")

# Pass callback to optimize
net.optimize(solver_name="highs", extra_functionality=flow_gate_constraint)
```

The pattern is clean:
- **Documented:** `extra_functionality` is in PyPSA's public API docs
- **No forking:** constraint injection via callback, not source modification
- **Shadow prices:** accessible via `net.model.constraints["flow_gate_upper"].dual`
- **Composable:** multiple callbacks can be chained
