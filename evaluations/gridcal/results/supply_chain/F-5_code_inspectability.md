# F-5: Code Inspectability (DCPF Execution Path)

## Finding

The DCPF execution path is fully traceable through 4 abstraction layers of pure Python code, from the API entry point to the sparse linear solver call. All formulation logic is readable and auditable. The only opaque step is the final `spsolve` call into scipy's compiled C/Fortran sparse solver.

## Evidence

**DCPF execution path (4 layers):**

1. **API layer:** `vge.power_flow(grid, options)` in `VeraGridEngine/api.py`
   - Creates `PowerFlowDriver`, calls `driver.run()`
   - Pure Python, well-documented function signature

2. **Driver layer:** `PowerFlowDriver.run()` in `Simulations/PowerFlow/power_flow_driver.py`
   - Compiles `NumericalCircuit` from the `MultiCircuit` data model
   - Dispatches to `multi_island_pf()` worker based on solver type
   - Pure Python orchestration

3. **Worker layer:** `multi_island_pf()` in `Simulations/PowerFlow/power_flow_worker.py`
   - Handles island detection and per-island dispatch
   - Routes `SolverType.Linear` to `linear_pf()` (linearized_power_flow.py)
   - Pure Python

4. **Solver layer:** `linear_pf()` in `Simulations/PowerFlow/NumericalMethods/linearized_power_flow.py`
   - Constructs reduced susceptance matrix (Bpqpv)
   - Computes power injection: `Pinj = Sbus[no_slack].real - (Bref @ Va_ref) * Vm[no_slack] + Pps[no_slack]`
   - Solves: `Va[no_slack] = linear_solver(Bpqpv, Pinj)` where `linear_solver` is `scipy.sparse.linalg.spsolve`
   - Reconstructs voltage: `V = polar_to_rect(Vm, Va)`
   - Computes branch flows from solved voltages
   - ~80 lines of readable Python with clear variable names

**Key formulation details visible in source:**
- Phase shifter correction term (`Pps = Bf.T @ tau`) is included
- ZIP load model support via `compute_zip_power(S0, I0, Y0, Vm)`
- Convergence check: `norm_f = np.linalg.norm(mismatch, np.inf)`
- B-matrix construction uses `get_sparse_type()` / `get_linear_solver()` abstraction for solver backend selection

**Opaque steps:**
1. `linear_solver(Bpqpv, Pinj)` — resolves to `scipy.sparse.linalg.spsolve`, a compiled C/Fortran SuperLU sparse direct solver. The algorithm is well-documented in scipy and academic literature but the compiled implementation is not source-auditable.
2. Numba JIT-compiled functions in sparse matrix utilities (`Utils/Sparse/csc_numba.py`) — the Python source is visible but runtime machine code is generated by LLVM and is opaque.

**Positive inspectability characteristics:**
- No obfuscation, minification, or bytecode-only distribution
- All `.py` source files are included in the PyPI sdist
- Variable names are descriptive (`Bpqpv`, `Pinj`, `Va`, `Vm`, `Sbus`)
- MPL-2.0 headers on all source files with SPDX identifiers

## Implications

VeraGridEngine achieves excellent code inspectability for its core power flow logic. An auditor can trace the complete DCPF formulation from API call to matrix solve in ~200 lines of readable Python across 4 files. The only non-inspectable components are standard scientific computing libraries (scipy, numpy) that are widely trusted and independently audited. The numba JIT compilation is a minor inspectability concern — the source is visible but the executed code is runtime-generated machine code.
