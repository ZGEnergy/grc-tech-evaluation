# F-5: Code Inspectability

## Result: PASS

## Finding

The execution path from API call to solver invocation is fully traceable through 4 layers of pure Python code. All formulation logic is readable and auditable. The only opaque steps are (1) the final `scipy.sparse.linalg.spsolve` call into compiled C/Fortran SuperLU and (2) numba JIT-compiled functions that produce runtime machine code from visible Python source.

## Evidence

**DCPF execution path (4 layers, all pure Python):**

1. **API layer:** `vge.power_flow(grid, options)` in `VeraGridEngine/api.py`
   - Creates `PowerFlowDriver`, calls `driver.run()`
   - ~10 lines, well-documented function signature

2. **Driver layer:** `PowerFlowDriver.run()` in `Simulations/PowerFlow/power_flow_driver.py`
   - Checks engine type (VeraGrid, NewtonPA, Bentayga, PGM, GSLV) with fallback to VeraGrid
   - Dispatches to `multi_island_pf()` for native engine
   - Pure Python orchestration

3. **Worker layer:** `multi_island_pf()` in `Simulations/PowerFlow/power_flow_worker.py`
   - Compiles `NumericalCircuit` from `MultiCircuit` data model
   - Handles island detection and per-island dispatch
   - Routes `SolverType.Linear` to `linear_pf()`

4. **Solver layer:** `linear_pf()` in `Simulations/PowerFlow/NumericalMethods/linearized_power_flow.py`
   - Constructs reduced susceptance matrix (Bpqpv)
   - Computes power injection with phase-shifter correction: `Pinj = Sbus[no_slack].real - (Bref @ Va_ref) * Vm[no_slack] + Pps[no_slack]`
   - Solves sparse linear system: `Va[no_slack] = linear_solver(Bpqpv, Pinj)`
   - Reconstructs voltage: `V = polar_to_rect(Vm, Va)`
   - ~80 lines of readable Python

**Linear OPF execution path (3 layers):**

1. **API:** `vge.linear_opf(grid, options)` -> `OptimalPowerFlowDriver(grid, options).run()`
2. **Driver:** Routes to PuLP-based formulation in `Formulations/linear_opf_ts.py`
3. **Solver:** PuLP dispatches to HiGHS via `highspy` bindings

**Module count in execution path:** 8 modules for DCPF, 10 for linear OPF (including PuLP interface)

**Opaque steps (2):**
1. `scipy.sparse.linalg.spsolve` -- compiled C/Fortran SuperLU sparse direct solver. Well-documented algorithm but compiled implementation is not source-auditable at the binary level.
2. Numba `@njit`-decorated functions in 38 files -- Python source is visible and auditable, but runtime machine code is generated by LLVM and is opaque.

**Positive inspectability characteristics:**
- No obfuscation, minification, or bytecode-only distribution
- All `.py` source files included in PyPI sdist
- Variable names are descriptive (`Bpqpv`, `Pinj`, `Va`, `Vm`, `Sbus`)
- MPL-2.0 headers with SPDX identifiers on all source files
- Consistent driver pattern across all analysis types

## Implications

VeraGridEngine achieves excellent code inspectability for its core power flow and optimization logic. An auditor can trace the complete DCPF formulation from API call to matrix solve in ~200 lines of readable Python across 4 files. The custom interior-point solver for AC OPF is also fully implemented in Python (not delegated to an external NLP solver like Ipopt), meaning the entire AC OPF formulation is inspectable. The only non-inspectable components are standard scientific computing libraries (scipy, numpy, HiGHS) that are widely trusted and independently audited.
