# C-4: SCUC 24hr on SMALL with HiGHS and SCIP

## Result: QUALIFIED PASS

## Approach

Scaled A-5 (SCUC on TINY) to the SMALL network (ACTIVSg 2000-bus, 544 generators, 3206
branches, 1125 loads). Since the SMALL network has no augmented timeseries data, a synthetic
24-hour load profile was generated by scaling base-case loads by hourly shape factors
(0.57--1.00 representing a typical winter weekday profile).

**Method: Sequential snapshot OPF** (each of 24 hours solved independently). The native
time-series OPF driver could not be used for two reasons:

1. **TapPhaseControl enum bug** (v5.6.28): The time-series OPF driver crashes with
   `ValueError: 0 is not a valid TapPhaseControl` on networks with transformers. This is
   the same bug documented in B-4. [tool-specific: TapPhaseControl enum validation error]
2. **Monolithic MILP intractability**: Even if the bug were fixed, the time-series driver
   formulates the 24-hour SCUC as a single MILP with ~13,056 binary variables (544 gens x
   24 hours). HiGHS ran for >25 minutes without producing a result before being terminated.
   [mixed: tool formulates monolithic MILP; solver cannot solve at scale]

Each hour was solved via `run_linear_opf_ts(grid, time_indices=None, solver_type=...)` in
snapshot mode, using the documented public API.

**Multi-threading note:** GridCal's PuLP-based OPF interface does not expose solver thread
configuration. The `run_linear_opf_ts` function signature has no thread parameter, and the
internal `lp_model.solve()` call passes no thread count to PuLP. HiGHS supports multi-threaded
MILP solving, but GridCal's binding does not expose this capability. All timings below are
single-threaded. [mixed: tool binding does not expose solver's multi-threading]

## Workarounds

- **What:** Used sequential snapshot OPF (each hour solved independently) instead of the
  native time-series OPF driver with UC mode.
- **Why:** Two compounding issues: (1) TapPhaseControl enum bug crashes the time-series
  driver on networks with transformers, and (2) the monolithic 24h MILP is computationally
  intractable for 544 generators.
- **Durability:** STABLE -- the sequential approach uses `run_linear_opf_ts` with
  `time_indices=None`, a documented public API function. The snapshot mode is well-tested
  and unlikely to break.
- **Grade impact:** The workaround loses inter-temporal UC coupling (min up/down times,
  ramp rates are not enforced across hours). This means the test demonstrates that 24
  independent hourly OPF solves complete successfully on the SMALL network, but not that
  true multi-period SCUC works at scale. This is a meaningful limitation.

from VeraGridEngine.Simulations.OPF.Formulations.linear_opf_ts import run_linear_opf_ts

for t in range(24):

        ld.P = float(ld.P * LOAD_SHAPE[t])

        grid=grid,
