## Executive Summary

This sweep analyzed v10 evaluation results across all six tools against the v10 protocol and rubric, identified systematic deficiencies in test design, network selection, and scoring consistency, and proposes 17 changes (PC-01 through PC-17) for incorporation into v11. Eight targeted probes were executed to resolve extraordinary claims and scoring ambiguities. The most consequential structural finding is that the C-SMALL gate conflates MILP capability with LP/power-flow scalability: four tools (pypsa, pandapower, matpower, powermodels) have seven to eight MEDIUM-tier tests suppressed by a gate that fires on SCUC failure regardless of whether SCUC is within the tool's scope. A second high-signal finding is that the ACTIVSg10k MEDIUM network is uncongested, making C-8 SCOPF MEDIUM vacuous for all three tools that executed it — the test measures solver robustness on a problem with no meaningful security constraints to enforce. Six additional high-signal themes spanning SCUC behavioral verification, SCOPF network quality, PSS/E format coverage, GridCal's soft constraint formulation, SCIP license misclassification in PowerModels, and qualified_pass severity ambiguity collectively motivate the bulk of the v11 redesign. Probe-010 confirmed a material license error: SCIP_jll v0.2.1 wraps SCIP 8.0.0, which is ZIB Academic — the Apache 2.0 switch occurred at 8.0.3, not 8.0.0, making the F-8 supply chain upgrade for PowerModels unwarranted. Probe-016 debunked the matpower exit-flag-mapping-bug theory: the GLPK failure is GLP_ETMLIM (time limit, no feasible solution) rather than GLP_EMIPGAP, and the SCUC is genuinely infeasible. No GitHub issues with the `protocol` label were open during the sweep period.

## Low-Signal Tests

### G-1, G-2, G-3: Network Ingestion Gates

**Signal level:** Low
**Outcome:** Universal pass across all six tools (6/6 pass at every tier)
**Root cause:** The tests verify element counts after loading standard MATPOWER .m files. Any tool that has reached evaluation-ready maturity passes trivially. The only informative outcome is a failure (broken importer).
**Proposed action:** PC-14 — tag as `test_category: gate_minimum_bar`; exclude pass outcomes from cross-tool pass rate numerator and denominator. The gates remain as preconditions but do not inflate apparent pass rates. Optional enhancement: add a data quality sub-check (cost function completeness) to provide discriminative signal. Source: pypsa-F11, pandapower-F09, gridcal-F12, powersimulations-F15, matpower-F07.
**v11 treatment:** Retained as gates, excluded from graded statistics.

### B-8: Reference Bus Configurability

**Signal level:** Low
**Outcome:** Universal pass across all six tools
**Root cause:** The pass condition requires "LMP values change consistently across configurations" when the slack bus is moved. In standard DC OPF, LMPs are invariant to reference bus choice — the angle reference cancels out of all KCL dual variables. Any correct DC OPF produces zero LMP variation across slack configurations, which satisfies the condition trivially. The test verifies API configurability only, not meaningful LMP sensitivity. Source: pypsa-F01 (zero LMP spread across all three configs), matpower-F06 (uniform shift, std dev $0.00).
**Proposed action:** PC-11 — redesign pass condition to test API configurability explicitly; remove LMP variation requirement for DC OPF.
**v11 replacement:** B-8 redesigned per PC-11.

### C-5: ACPF Progressive Relaxation

**Signal level:** Low
**Outcome:** Universal pass; no binding constraints at any relaxation level
**Root cause:** The ACTIVSg2000 SMALL and ACTIVSg10k MEDIUM networks are lightly loaded. No thermal or voltage constraints bind at base loading, so progressive relaxation (0%, 10%, 20%) has no effect on the ACPF solution. The test confirms convergence (which passes for all tools) but provides no signal about constraint-handling capability. Source: matpower-F08 (0 of 3,206 SMALL branches binding), powersimulations-F14 (thermal relaxation does not affect NR equations).
**Proposed action:** No assigned PC number; recommend running C-5 on a network with pre-confirmed binding constraints or increasing loading until at least one constraint is active.
**v11 treatment:** Retained pending network selection fix; no scoring change.

### B-6: Architecture Audit

**Signal level:** Low
**Outcome:** Universal pass; no runtime execution
**Root cause:** B-6 is a static source code and documentation review. It produces accurate architectural findings but does not differentiate tools in any runtime-measurable way. Pass is effectively guaranteed for any tool with documented layered architecture. Source: pm-F11.
**Proposed action:** No PC assigned; retain as qualitative signal but do not weight equally with runtime extensibility tests in grade calculations.
**v11 treatment:** Retained; noted in rubric as qualitative-only dimension.

### D-1: Install-to-First-Solve Timing (MATPOWER)

**Signal level:** Low (for MATPOWER specifically)
**Outcome:** MATPOWER records `timing_source: estimated`, `wall_clock_seconds: null`
**Root cause:** The protocol requires stopwatch timing from a clean devcontainer state. MATPOWER's D-1 provides a retrospective estimate rather than a measured value, making cross-tool timing comparison invalid. Source: matpower-F10.
**Proposed action:** PC-15 — require actual measured timing; results with `timing_source: estimated` flagged as incomplete.
**v11 treatment:** Require measured timing in all D-1 result files.

### Probe Details

#### probe-001: PyPSA G-FNM-3 Zero-Deviation Claim

**Classification:** claim_debunked (weak)
**Finding:** The G-FNM-3 result reports 0.0 mean and max deviation on 27,862 buses and 32,532 branches. Probe-001 re-extracted deviations at full float64 precision and found: max bus angle deviation = 1.073e-8 degrees, max branch flow deviation = 5.76e-7 percent. These values round to 0.000000 at 6 decimal places, producing the display artifact. The deviations are floating-point noise at the limit of float64 precision for values of this scale — not evidence of a formulation difference.
**PASS grade:** Confirmed correct. All buses pass the 1.0-degree tolerance by a factor of ~100 million.
**Validation report discrepancy:** The validation-report.md lists G-FNM-3 as FAIL with reason "systematic impedance conversion differences via MATPOWER fallback." This is a stale artifact from before the shared matpower_loader branch-status patch was applied. The individual result file (status: pass) is the authoritative source.
**Impact on v11:** PC-08 — require scientific notation (:.6e format) for deviation values in G-FNM result files to distinguish float64 noise from true machine-zero agreement.

#### probe-003: pandapower A-3 Branch Shadow Prices

**Classification:** claim_supported (sweep concern refuted)
**Finding:** The sweep raised concern that pandapower's 46/46 non-zero shadow prices might be interior-point numerical artifacts. Probe-003 re-ran the A-3 DC OPF and confirmed: minimum shadow price = 8.79 $/MWh, Pearson correlation with loading% = 0.875, all 46 branches exceed 1.0 $/MWh. Interior-point artifacts appear at 1e-8 to 1e-4 $/MWh — three to six orders of magnitude below the observed minimum.
**Root cause:** With 70% branch derating on the 39-bus network, the entire network is congestion-coupled. Relaxing any branch limit allows cheaper generation to dispatch, producing economically significant shadow prices on all constraints even for lightly loaded branches. This is correct LP/QP behavior, not an artifact.
**Impact on v11:** No protocol change required. Pandapower A-3 outcome confirmed.

#### probe-005: GridCal DCOPF Soft Constraints

**Classification:** confirmed_issue
**Finding:** Probe-005 inspected GridCal's `VeraGridEngine/Simulations/OPF/opf_driver.py` and confirmed explicit LP slack variables `flow_slacks_pos` and `flow_slacks_neg` per branch. Runtime verification: branch 2_3_1 reaches 103.5% loading (362 MW vs 350 MW derated limit) in the optimal solution, with overload slack = -12.22 MW. Only soft constraints allow a branch to exceed its rated limit in the optimal LP solution.
**Impact on A-3:** The three pass conditions (convergence, LMP extractability, binding branch count) do not verify hard-limit enforcement. Under a corrected condition (`max(loading_percent) <= 1.0 + epsilon`), GridCal A-3 = fail.
**Impact on v11:** PC-10 — add hard constraint enforcement verification to A-3 pass condition. Soft-constraint DCOPF should be classified as partial_pass with penalty coefficient documented.

#### probe-007: GridCal G-FNM-3 Branch Deviation Pattern

**Classification:** classification_plausible_with_caveats
**Finding:** The 326 outlier branches are 88.7% transformer-adjacent (289/326), meeting the 80% structural-pattern threshold for the formulation_difference classification. However, four of the top five deviating branches are typed as "Line" (not transformer), and deviation magnitudes reach 562,955% — far beyond what tap-ratio mishandling alone would produce. The protocol has no magnitude cap for formulation_difference classifications, allowing arbitrarily large deviations to be excused.
**Assessment:** The qualified_pass verdict is defensible as a protocol outcome, but understates severity. GridCal DCPF branch flows for transformer-adjacent branches in real grid models are unreliable at the operational level (errors up to ~3000×).
**Impact on v11:** Add a maximum permissible deviation cap (e.g., 1000%) to the formulation_difference classification in pass_conditions.json. The classification_plausible_with_caveats finding does not require a grade change but does require a protocol design fix.

#### probe-009: PowerModels A-9 Benders Convergence

**Classification:** inconclusive
**Finding:** No code was executed — the A-9 result file already transparently documents all relevant facts. The evaluator explicitly states N-1 infeasibility and 1-iteration behavior. The Benders API mechanism (PTDF/LODF computation, constraint injection, re-solve) is genuinely demonstrated. However, the key question — does the iterative algorithm converge to a security-constrained dispatch when a feasible SCOPF solution exists? — is never answered because the IEEE 39-bus network is N-1 infeasible under the Modified Tiny load profile.
**Assessment:** The qualified_pass is defensible as a mechanism verification. The test design gap (no feasible SCOPF scenario provided) means no tool demonstrated multi-iteration Benders convergence during this evaluation.
**Impact on v11:** PC-16 — pre-compute and publish a feasible N-1 SCOPF configuration for case39. Pass condition should require multi-iteration convergence OR explicit feasibility demonstration.

#### probe-010: PowerModels SCIP License

**Classification:** claim_supported (F-3 is correct; F-8 contains a material error)
**Finding:** SCIP_jll v0.2.1+0 wraps SCIP 8.0.0 (confirmed via JuliaBinaryWrappers release tarball URL: `scipoptsuite-8.0.0.tgz`, build date April 28, 2022). The Apache 2.0 license switch happened at SCIP 8.0.3 (December 2022). F-8's assertion that "SCIP v8.0 switched to Apache 2.0" is factually incorrect — the switch was at 8.0.3, not 8.0.0. `SCIP.SCIPversion()` returning "8.0" indicates SCIP 8.0.0, which is ZIB Academic.
**Supply chain implication:** Any commercial deployment using the pinned manifest (SCIP_jll v0.2.1) requires either a SCIP commercial license or exclusion of SCIP from the solver stack. F-8's supply chain upgrade to P is unwarranted; the correct outcome is QP.
**Broader implication:** JLL binary artifact licenses (ZIB Academic) differ from Julia wrapper package licenses (MIT). The evaluation skill must check both, with the binary license governing.
**Impact on v11:** PC-13 — correct F-8 outcome for PowerModels to QP; update skill to record both wrapper and binary artifact licenses for JLL packages.

#### probe-013: PowerSimulations A-2 Convergence Diagnostics

**Classification:** claim_debunked
**Finding:** The evaluation set `global_logger(ConsoleLogger(stderr, Logging.Error))`, suppressing @info messages. The PowerFlows.jl source (`powerflow_method.jl` line 314) emits `@info("The $T solver converged after $i iterations.")` on success and `@error(...)` on failure. Enabling Info-level logging captures the iteration count (1 for IEEE 39-bus). Additionally, `solve_powerflow` returns `missing` on non-convergence — a returned `Dict{String, DataFrame}` is a binary convergence guarantee stronger than voltage-profile inference.
**Power balance verification:** IEEE 39-bus ACPF: 6,297.9 MW generation, 6,254.2 MW load, 43.6 MW losses (0.69%) — physically consistent.
**Assessment:** The A-2 qualified_pass annotation overstates the tool limitation. Convergence IS verifiable via logging and return type. The residual value remains inaccessible (correct claim), but iteration count and binary convergence are available.
**Impact on v11:** PC-04 — formalize a tiered convergence evidence hierarchy: (1) residual value, (2) iteration count via any API including logging, (3) binary convergence indicator via return type, (4) voltage profile proxy. Skill should enable Julia @info logging during ACPF solves.

#### probe-016: MATPOWER GLPK Exit-Flag Bug

**Classification:** claim_debunked
**Finding:** The matpower-F01 finding claimed GLPK exit code errnum=9 = GLP_EMIPGAP (MIP gap reached), implying the SCUC solved successfully but the result could not be extracted. Probe-016 established that errnum=9 in Octave 8.4.0 = GLP_ETMLIM (time limit reached), not GLP_EMIPGAP (which is errnum=14). The output confirms: `mdo.QP.output.status = -1` (no feasible solution found), objective value = NA, non-zero variables = 0. No feasible integer solution was ever found.
**One-line fix claim:** Refuted. `miqps_glpk.m` already contains the dead-code path `(errnum == 9 && extra.status == 2)` for the case where a feasible solution exists at time limit. This path requires `status=2` (feasible), which is not present. Since no feasible solution exists, fixing exit flag mapping would not recover any result.
**Impact on v11:** The C-SMALL gate cascade to MEDIUM tests remains appropriate for matpower — GLPK genuinely cannot solve the case39 SCUC to feasibility. However, this confirms the gate design issue (T-01, PC-01): MATPOWER's strong LP/PF capability at scale is suppressed by a gate that fires on a genuine SCUC limitation.

## Proposed Changes

### Test Redesigns

**PC-01 — Decouple LP/PF MEDIUM tests from the MILP C-SMALL gate** (High priority)
The C-SMALL gate currently blocks all MEDIUM-tier tests when C-4 (SCUC SMALL) fails, regardless of whether the failure is scope-based (SCUC absent), MILP-solver-based (timeout), or genuine infeasibility. LP/PF MEDIUM tests (C-1 DCPF, C-2 ACPF, C-3 DC OPF, C-9 PTDF, C-10 distributed slack) are independent of MILP capability and should run unconditionally. C-8 SCOPF (which depends on LP, not MILP SCUC) should be gated on C-3, not C-4.
Evidence: pypsa-F09, pandapower-F01, matpower-F01/probe-016, pm-F13. Four tools affected.

**PC-03 — Replace ACTIVSg10k with a congested MEDIUM network for C-8 SCOPF** (High priority)
The ACTIVSg10k MEDIUM network has a maximum branch loading of ~84.72% at base case. No N-1 contingency creates post-contingency violations, making C-8 MEDIUM a vacuous test of solver robustness rather than SCOPF expressiveness. All three tools that ran C-8 MEDIUM confirmed zero meaningful redispatch independently (gridcal-F02: 2.67e-7 MW max dispatch change; pm-F02: 1 Benders iteration, non-converged; powersimulations-F09: HiGHS OTHER_ERROR crash).
Options: (A) Load-scale ACTIVSg10k to ~120% until 5% of branches exceed 95% loading; (B) Use ACTIVSg2000 SMALL as the SCOPF benchmark network (confirmed congestion at C-8 SMALL level); (C) synthetic stressed ACTIVSg10k with reduced thermal ratings on critical paths. Pass condition must include minimum redispatch requirement (≥5 MW aggregate vs base DCOPF).

**PC-09 — Separate PSS/E format support from power-system capability in FNM suite** (High priority)
Zero of six evaluated tools successfully ingested the FNM via the PSS/E v31 native path. All non-MATPOWER tools used the fnm_main_island.m MATPOWER fallback, making cross-tool G-FNM comparison conditional on a common preprocessor. G-FNM-1 failure (PSS/E format gap) should not cascade to block G-FNM-2 and G-FNM-3; field coverage and DCPF accuracy should always be assessed via the best available path. The FNM should be provided in both PSS/E and MATPOWER formats as named first-class inputs, not fallback alternatives.
Evidence: pypsa-F05, pandapower-F13, gridcal-F08, pm-F07. Four tools affected.

**PC-11 — Redesign B-8 to test API configurability rather than LMP variation** (Medium priority)
B-8's LMP variation requirement is mathematically vacuous for all correct DC OPF implementations — LMPs are invariant to reference bus choice. The test currently measures only API configurability (can the tool accept a different reference bus without error), which is a valid test but should be stated as such. Source: pypsa-F01, matpower-F06.

**PC-16 — Provide a feasible N-1 SCOPF test scenario for the TINY network** (Medium priority)
The IEEE 39-bus network under the Modified Tiny load profile is N-1 infeasible at nominal ratings, preventing any tool from demonstrating multi-iteration Benders or iterative SCOPF convergence. The pass condition should require multi-iteration convergence OR explicit demonstration that the joint N-1 problem is infeasible. Pre-computed feasible N-1 subset should be published in eval-config.yaml.
Evidence: pypsa-F04, pm-F03/probe-009, powersimulations-F04. Three tools affected.

### New Tests

**PC-12 — Add cross-reference independence check to G-FNM-3** (Low priority)
MATPOWER generates the G-FNM-3 reference DCPF solution from its own format using its own solver, making MATPOWER's zero-deviation result mathematically guaranteed. A bus-injection power balance check (tool-independent necessary condition for solution validity) or cross-comparison against an independently computed DCPF would validate the reference for transformer branches. Probe-001 provides indirect validation for line-dominated segments (PyPSA deviations at float64 noise level), but transformer branch correctness of the reference remains unvalidated.
Evidence: matpower-F02, probe-001.

### Removed Tests

No tests are proposed for removal. Gate tests G-1/G-2/G-3 are retained as preconditions but excluded from graded statistics (PC-14).

### Scoring Changes

**PC-07 — Add severity tiers to qualified_pass** (High priority)
The qualified_pass status is applied to results spanning stable workarounds (low friction), blocking architectural impossibilities, non-converged SCOPF runs, and solver crashes. The workaround_class field documents severity, but status aggregation collapses all into one opaque outcome. Proposed three-tier replacement:
- `qualified_pass`: stable workaround, full capability demonstrated, low friction (current meaning)
- `partial_pass`: non-trivial workaround, some criterion unmet, moderate friction; maps to workaround_class: fragile or blocking
- `constrained_pass`: capability demonstrated under constraints that affect generalizability (uncongested network, single-threaded solver, simplified formulation)

Specific reclassifications: PyPSA A-11 (blocking workaround class: should be partial_pass or fail, not qualified_pass); PowerModels C-8 MEDIUM (non-converged: should be constrained_pass); PowerSimulations C-8 MEDIUM (solver crash: should be fail with context); PowerModels A-12 (three simultaneous workarounds: should be qualified_pass, currently awarded pass).
Evidence: pypsa-F12, pm-F02, pm-F06, powersimulations-F09, pandapower-F10. Five tools affected.

**PC-13 — Fix SCIP license classification for PowerModels** (High priority)
F-3 correctly classifies SCIP_jll v0.2.1 as ZIB Academic (confirmed by probe-010). F-8 incorrectly claims Apache 2.0 — the switch happened at SCIP 8.0.3 (December 2022), not 8.0.0 (April 2022, as pinned). The F-8 supply chain P for PowerModels should be corrected to QP. Broader guidance: for Julia JLL packages, the binary artifact license (not the Julia wrapper license) governs deployment classification.
Evidence: pm-F08, probe-010.

**PC-14 — Exclude gate test pass outcomes from cross-tool pass rate comparisons** (Low priority)
All six tools pass G-1/G-2/G-3 in every evaluation. Including these in pass rate statistics uniformly inflates apparent capabilities by a fixed floor without providing any discriminative signal. Tag as `test_category: gate_minimum_bar`; exclude from graded statistics. Optional enhancement: add data quality sub-check (cost function completeness) that would provide discriminative signal if any tool fails.
Evidence: pypsa-F11, pandapower-F09, gridcal-F12, powersimulations-F15, matpower-F07. All six tools.

### Modified Tests

**PC-02 — Add min up/down binding verification to A-5 SCUC** (High priority)
The current 2-cycling-generator threshold is met by all tools attempting SCUC on the 39-bus TINY network, but does not verify that min up/down time constraints are actually binding. With a 1.18x capacity-to-load ratio, the optimizer can satisfy the cycling count through cost-driven shutdowns without the temporal constraints ever being active. Proposed addition: re-run with min_up_time=min_down_time=0 and compare commitment schedules — if no generator changes status, no temporal constraint was binding, which is a protocol failure condition. MIP gap must also be extractable and reported.
Evidence: pypsa-F02, pm-F09, powersimulations-F12, matpower-F05, gridcal-F04. Five tools affected.

**PC-04 — Formalize convergence verification hierarchy for A-2 and C-2** (Medium priority)
The protocol requires reporting the convergence residual, but at least three tools have no public API path to extract it. Probe-013 established that PowerSimulations.jl does emit the iteration count at @info level (suppressed by the original evaluation's logging configuration), and that the return type structurally guarantees convergence. The protocol should specify a tiered evidence hierarchy: (1) residual value; (2) iteration count via any API including logging; (3) binary convergence indicator via return type or exception; (4) voltage profile proxy as last resort. Evaluators should attempt all tiers and document which tier was achieved.
Evidence: pandapower-F02, pm-F05, powersimulations-F02, probe-013. Three tools affected.

**PC-05 — Add binding ramp constraint evidence requirement to A-6** (Medium priority)
RTS-GMLC technology-median ramp rates are 100-1000x larger than the dispatch changes driven by the 39-bus load profile. No tool's A-6 result shows binding ramp constraints under current parameters. The pass condition should require at least one ramp constraint with dual value > 0, with the generator index, hour, and dual magnitude reported. Proposed fix: scale ramp rates down by 10x in gen_temporal_params.csv to force binding.
Evidence: matpower-F04, pm-F13, powersimulations-F08. Three tools affected.

**PC-06 — Distinguish SCED-with-UC from ED-only when SCUC is absent** (Medium priority)
When a tool cannot perform SCUC, A-6 reduces to pure multi-period economic dispatch. Currently, full two-stage SCED and ED-only both receive qualified_pass, conflating genuinely different capability levels. Add explicit sub-categorization: `sced_mode: full_sced | ed_only | ed_with_security`. Only full_sced maps to pass; ed_only maps to fail with context.
Evidence: pm-F13, pandapower-F07, powersimulations-F08. Three tools affected.

**PC-08 — Require full float64 precision for deviation reporting in G-FNM-3** (Low priority)
The 6-decimal-place rounding in result files causes sub-1e-6 deviations to display as 0.000000. Probe-001 confirmed PyPSA's reported 0.0 bus angle deviation is actually 1.07e-8 degrees. Require scientific notation (:.6e format) for all deviation values. Pass/fail thresholds remain unchanged; only reporting format changes.
Evidence: pypsa-F06, matpower-F02, probe-001.

**PC-10 — Add hard constraint enforcement verification to A-3 DCOPF** (High priority)
The A-3 pass condition does not verify that branch thermal limits are enforced as hard constraints. Probe-005 confirmed GridCal's linear_opf uses soft branch flow constraints (LP slack variables), allowing branches to exceed rated limits in the optimal solution (branch 2_3_1 at 103.5%). Add to A-3 pass condition: `max(loading_percent) <= 1.0 + epsilon` (epsilon = 1e-4 for numerical tolerance). Soft-constraint DCOPF should be classified as partial_pass with penalty coefficient documented.
Evidence: gridcal-F06, probe-005.

**PC-15 — Standardize solver thread count reporting in scalability tests** (Medium priority)
PowerSimulations C-4 records 404 seconds for SCUC with HiGHS at 1 thread on a 32-core machine. Without thread normalization, cross-tool MILP timing comparisons are invalid. Add `cpu_threads_used` and `cpu_threads_available` to all C-suite result frontmatter. For MILP tests (C-4, C-8), report both 1-thread and max-thread wall-clock if the solver supports parallel execution.
Evidence: powersimulations-F16, matpower-F10.

### Skill Updates

**PC-17 — Require binary commitment variable verification in SCUC skill** (Medium priority)
GridCal infers commitment from generator power output thresholding (gen_power > 0.1 MW) rather than reading binary commitment variables from the solver. If the solver runs an LP relaxation producing near-binary values, the inferred schedule appears valid but is not guaranteed MILP-generated. The skill should: (1) attempt to read binary commitment variables from the solver model; (2) if unavailable, extract MIP gap; (3) if MIP gap is unavailable, re-solve with LP relaxation and compare — if identical, flag as potential LP relaxation rather than true MILP solve.
Evidence: gridcal-F04, pm-F01.

## Deferred Items

The following items were identified during the sweep but are deferred to a future sweep or parallel work:

1. **PyPSA A-11 scoring correction** — pysa-F07 argues A-11 should be scored fail (not qualified_pass) because distributed slack DC OPF is architecturally impossible in the optimize() path and the AC PF workaround does not satisfy the pass condition. The scoring change is deferred pending the PC-07 severity tier implementation; once partial_pass and constrained_pass are defined, A-11 should be reclassified as partial_pass (blocking workaround class) rather than qualified_pass.

2. **PowerModels G-FNM-3 DCMPPowerModel retest** — pm-F04 identifies that the test mandates solve_dc_pf (which internally uses DCPPowerModel with simplified B-matrix), when DCMPPowerModel (full B-matrix including taps) would likely produce a passing result on the transformer-heavy FNM. The protocol should be clarified to permit the most appropriate formulation, not hardcode the function call. Deferred pending G-FNM-3 redesign work.

3. **GridCal E-7 adoption claims** — gridcal-F11 identifies that claimed operational adoption by Redeia, Schneider Electric, and GE Vernova cannot be verified from public sources. A follow-up verification step (direct outreach or public procurement records) was identified but not pursued during this sweep.

4. **PowerSimulations C-4 multi-threaded timing** — powersimulations-F16 documents 404 seconds at 1 thread on a 32-core machine. A multi-threaded re-run (32 threads, same HiGHS version) would provide practical deployment timing. Deferred pending protocol standardization of thread count reporting (PC-15).

5. **PyPSA A-10 LMP decomposition completeness** — pypsa-F03 identifies that the three-component LMP decomposition check (energy + congestion + loss = total LMP) was performed as a two-component check. A full three-component verification is warranted but was not probed due to prioritization of higher-signal items.

6. **PowerModels GLPK C-3 speedup explanation** — pm-F10 documents an unexplained 5x speedup between v9 (300s+ timeout) and v10 (61.86s convergence) on GLPK for MEDIUM DC OPF. No definitive explanation was found; JIT warm-up and cost linearization differences are candidate explanations. Deferred.
