# Sweep Findings

:::info[Protocol Context]
These findings originate from the v4-to-v5 systematic sweep. They have been incorporated into the v10 grading and are preserved here for traceability.
:::

## Executive Summary

Between the v4 and v5 protocol revisions, a systematic sweep re-evaluated every scored result across all six tools, identifying patterns of shared failure, inconsistent grading, and structural gaps in the test protocol itself. This sweep produced **13 cross-cutting themes** -- recurring issues that affect multiple tools or multiple rubric dimensions simultaneously. Rather than isolated scoring corrections, these themes expose systemic weaknesses in the evaluation infrastructure: networks that lack discriminative power, timings that were estimated rather than measured, and protocol specifications that are structurally unachievable.

Three clusters proved most impactful. **Network insufficiency** (T01--T03) affects 20+ individual test results: the ACTIVSg10k network has no binding branch constraints, IEEE case39 has uniform generator costs, and SCUC on case39 produces zero cycling -- meaning a substantial fraction of expressiveness and scalability tests at TINY and MEDIUM scale measure only formulation existence, not correctness or performance. **Estimated timings** (T04--T05) undermine scalability grades directly: multiple tools received scores based on projected or cascaded failures rather than measured execution. **The PTDF phase-shifter correction gap** (T06) invalidated every Extensibility B-9 result for tools using the standard PTDF formula, since ACTIVSg10k's five phase-shifting transformers produce correction terms up to 743 MW that the standard formula omits entirely.

These 13 themes drove concrete v5 protocol changes: mandatory network preprocessing steps, the `blocked_by` cascaded-failure annotation, replacement of the lossy DCOPF reference validation with internal consistency checks, and tightened requirements around measured (not estimated) wall-clock timings.

## Themes

### T01: ACTIVSg10k Has No Binding Constraints {#t01}

<details className="eval-details" open>
<summary>
T01: ACTIVSg10k Has No Binding Constraints
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#e3f2fd', color: '#1565c0'}}>Network Insufficiency</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffcdd2', color: '#b71c1c'}}>HIGH</span>
</summary>

**Affected Tools:** 5 | **Evidence:** 5 findings

The MEDIUM-scale network (ACTIVSg 10k-bus) produces zero binding branch constraints in DCOPF across all tools. Maximum branch loading reaches only 84--85%, and LMPs are perfectly uniform across all 10,000 buses. This means that every OPF-based test at MEDIUM scale -- including A-3 (DCOPF), A-9 (SCOPF), A-11 (multi-period OPF), B-8 (custom constraints), C-3 (scalability DCOPF), and C-10 (scalability SCOPF) -- measures only whether the tool can formulate and solve the problem, not whether it produces economically meaningful results.

Probe-007 verified that pandapower's LMPs on ACTIVSg10k are uniform to machine precision (standard deviation = 6e-12 $/MWh). When artificial congestion is introduced by reducing a single branch capacity, meaningful LMP spreads of 16.3--24.4 $/MWh appear immediately, confirming that the tools are capable of producing differentiated prices but the network simply does not require it.

The v5 protocol addresses this by adding a mandatory congestion verification step and providing alternative congested networks for tests that require binding constraints to be discriminative.

**Proposed Remedy:** Replace or augment ACTIVSg10k with a network that has binding constraints under normal dispatch, or add a mandatory preprocessing step that introduces artificial congestion.

**Related Probes:** [probe-007](./probe-results#probe-007)

</details>

### T02: case39 Uniform Costs and No Congestion {#t02}

<details className="eval-details">
<summary>
T02: case39 Uniform Costs and No Congestion
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#e3f2fd', color: '#1565c0'}}>Network Insufficiency</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffe0b2', color: '#e65100'}}>MEDIUM</span>
</summary>

**Affected Tools:** 5 | **Evidence:** 5 findings

IEEE 39-bus has identical generator cost coefficients (c1 = 0.3 $/MWh for all 10 generators) and no binding branch constraints, producing perfectly uniform LMPs across all 39 buses for every tool tested. This makes TINY-scale OPF tests purely a formulation existence check: any tool that can set up and solve the LP will produce the same answer.

At TINY scale, A-3 (DCOPF) confirms convergence but cannot distinguish LMP quality. A-9 (SCOPF) produces an objective function equal to the unconstrained OPF objective, providing zero cost-premium signal. The SMALL network (ACTIVSg 2000-bus) has heterogeneous costs and does produce meaningful LMP spreads, confirming that the issue is network-specific rather than methodological.

**Proposed Remedy:** Add heterogeneous cost coefficients to the TINY reference case, or supplement with a second TINY network that has cost diversity and at least one binding constraint.

</details>

### T03: SCUC Produces No Generator Cycling on case39 {#t03}

<details className="eval-details">
<summary>
T03: SCUC Produces No Generator Cycling on case39
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#e3f2fd', color: '#1565c0'}}>Network Insufficiency</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffe0b2', color: '#e65100'}}>MEDIUM</span>
</summary>

**Affected Tools:** 3 | **Evidence:** 3 findings

Every tool capable of SCUC (PyPSA, PowerSimulations, MATPOWER) reports all 10 generators committed for all 24 hours with zero startups or shutdowns. The capacity-to-load ratio of 7,367 MW available vs 6,254 MW peak demand, combined with startup costs, means that decommitting any generator is never optimal. MATPOWER confirms that ramp utilization is uniform at 26.7% across all generators, meaning ramp constraints are never close to binding either.

This makes A-5 a formulation existence test -- it verifies that the tool can express a SCUC problem with minimum up/down time, startup costs, and ramp constraints, but it cannot verify that the tool handles cycling transitions correctly. A network with tighter capacity margins or asymmetric startup costs would be needed to test actual commitment switching behavior.

**Proposed Remedy:** Provide a supplementary TINY network with a capacity margin narrow enough to force at least one generator to cycle during the 24-hour horizon.

</details>

### T04: Estimated Timings Without Execution {#t04}

<details className="eval-details">
<summary>
T04: Estimated Timings Without Execution
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#fff3e0', color: '#e65100'}}>Scoring Inconsistency</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffcdd2', color: '#b71c1c'}}>HIGH</span>
</summary>

**Affected Tools:** 3 | **Evidence:** 6 findings

PowerSimulations has four scalability tests with null `wall_clock_seconds` and estimated language in the result narrative. PowerModels scored two tests as FAIL without execution, based on projected infeasibility extrapolated from smaller-scale runs. The protocol explicitly requires measured timings: "Record everything. For each test, record: Wall-clock time."

Probe-020 confirmed that PowerSimulations' C-3 was never executed -- the first attempt exceeded the 600s timeout (which includes JIT compilation overhead), and subsequent results were estimated from incomplete data. Probe-016 confirmed that PowerModels' projections were directionally correct (SCOPF with just 5 contingencies timed out at 567s, validating the claim that 500 contingencies is infeasible), but the scoring was still based on extrapolation rather than measurement.

The distinction matters because estimated timings cannot capture solver-specific behavior at scale (e.g., memory exhaustion, numerical instability, or non-linear scaling). A tool projected to fail might succeed with a different solver configuration, while a tool projected to succeed might hit an unexpected bottleneck.

**Proposed Remedy:** Require all scalability grades to include measured `wall_clock_seconds`. Tests that exceed the timeout should be scored as TIMEOUT with the measured partial timing, not estimated as FAIL.

**Related Probes:** [probe-020](./probe-results#probe-020), [probe-016](./probe-results#probe-016)

</details>

### T05: Cascaded Failures Inflate Fail Counts {#t05}

<details className="eval-details">
<summary>
T05: Cascaded Failures Inflate Fail Counts
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#fff3e0', color: '#e65100'}}>Scoring Inconsistency</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffe0b2', color: '#e65100'}}>MEDIUM</span>
</summary>

**Affected Tools:** 4 | **Evidence:** 8 findings

Multiple tools have scalability tests that fail solely because their prerequisite expressiveness test failed first. These cascaded failures add no new information about tool capability but inflate the apparent gap count, making some tools appear to have more deficiencies than they actually do.

For pandapower, 3 of 14 fails are cascaded (C-4, C-8, C-10 fail because the corresponding expressiveness tests A-5, A-9, A-11 failed). For GridCal, 2 of 16 fails are cascaded (C-4, C-6). The remaining fails for both tools are independent. The effect is modest in absolute terms but can shift relative rankings when tools are closely matched.

**Proposed Remedy:** Add a `blocked_by` field to result frontmatter that links a scalability test to its prerequisite expressiveness test. Cascaded failures should be reported separately from independent failures in summary statistics.

</details>

### T06: PTDF Phase-Shifter Correction Gap {#t06}

<details className="eval-details">
<summary>
T06: PTDF Phase-Shifter Correction Gap
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#fce4ec', color: '#c62828'}}>Protocol Gap</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffcdd2', color: '#b71c1c'}}>HIGH</span>
</summary>

**Affected Tools:** 3 | **Evidence:** 3 findings

ACTIVSg10k contains 5 phase-shifting transformers that create Pbusinj (bus injection correction) and Pfinj (branch flow correction) terms. The standard PTDF formula `flow = PTDF @ Pinj` omits these corrections, producing errors up to 743 MW on individual branches. The full equation is `flow = PTDF @ (Pinj - Pbusinj) + Pfinj`, and applying this correction eliminates ALL error to machine precision (residual < 1e-12).

The original B-9 (PTDF extraction) evaluations attributed the divergence to shunts or island connectivity issues. Probes 008, 010, and 012 demonstrated that shunts have zero MW impact and the network has exactly one island -- the entire error is caused by phase shifters. PowerModels handles the correction internally (error < 1e-11), while PyPSA, pandapower, and GridCal's external PTDF matrices expose the raw uncorrected values.

This is a protocol gap because B-9 did not specify whether the PTDF validation should account for phase-shifter corrections. A tool that exposes the mathematically standard PTDF matrix is not "wrong" -- the correction is a separate computation that the user must apply when phase shifters are present.

**Proposed Remedy:** Add phase-shifter correction terms to the B-9 validation procedure, or split B-9 into two sub-tests: basic PTDF accuracy on networks without phase shifters, and corrected PTDF on networks with phase shifters.

</details>

### T07: ACTIVSg10k Zero-Impedance Branch Inconsistency {#t07}

<details className="eval-details">
<summary>
T07: ACTIVSg10k Zero-Impedance Branch Inconsistency
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#e8f5e9', color: '#2e7d32'}}>Data Quality</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffe0b2', color: '#e65100'}}>MEDIUM</span>
</summary>

**Affected Tools:** 4 | **Evidence:** 4 findings

ACTIVSg10k has 2,462 branches with zero thermal rating and 3 transformers with zero reactance. Different tools handle these edge cases differently, producing contradictory pass/fail outcomes that reflect data interpretation choices rather than tool capability. PyPSA treats zero-capacity branches as binding constraints (infeasible), while other tools treat them as unconstrained (unlimited capacity).

This creates a paradox: PyPSA fails expressiveness A-3 at MEDIUM scale but passes scalability C-3 at MEDIUM after data fixes are applied. The same tool on the same network produces different outcomes depending on whether raw or cleaned data is used, making the result a test of data preprocessing rather than optimization capability.

**Proposed Remedy:** Standardize network preprocessing as an explicit protocol step: specify how zero-rating branches and zero-impedance transformers should be handled before any analysis begins.

</details>

### T08: Stochastic Perturbation Methodology Causes Excessive Infeasibility {#t08}

<details className="eval-details">
<summary>
T08: Stochastic Perturbation Methodology Causes Excessive Infeasibility
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#fce4ec', color: '#c62828'}}>Protocol Gap</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffe0b2', color: '#e65100'}}>MEDIUM</span>
</summary>

**Affected Tools:** 3 | **Evidence:** 3 findings

The B-4/C-6 stochastic scenario methodology perturbs both load and generator capacity simultaneously using independent random draws. This combination produces very high infeasibility rates -- pandapower on ACTIVSg2000 (SMALL) converges on only 0.42--2.1% of scenarios (probe-006), GridCal on SMALL converges on 47%, and PowerSimulations shows similar patterns.

Uniform load scaling alone (0.7x--1.1x of base load) converges reliably across all tools, confirming that the infeasibility is caused by the interaction of simultaneous perturbations: random load increases combined with random capacity decreases frequently produce scenarios where total demand exceeds available generation. This is a methodology issue, not a tool deficiency, since the infeasibility rate is similar across all tools.

**Proposed Remedy:** Decouple load and capacity perturbations, or constrain the perturbation space to ensure that total available capacity always exceeds total demand by a minimum reserve margin.

</details>

### T09: Lossy DCOPF MATPOWER Reference Validation Unachievable {#t09}

<details className="eval-details">
<summary>
T09: Lossy DCOPF MATPOWER Reference Validation Unachievable
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#fce4ec', color: '#c62828'}}>Protocol Gap</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffe0b2', color: '#e65100'}}>MEDIUM</span>
</summary>

**Affected Tools:** 4 | **Evidence:** 4 findings

The A-10 pass condition requires MATPOWER reference validation for lossy DC OPF, but MATPOWER's `rundcopf` is strictly lossless. MATPOWER can compute post-hoc loss estimates after solving, but these estimates are not part of the optimization -- they are applied after the fact and do not produce comparable LMPs. This makes the reference validation structurally impossible as currently specified.

All four tools that attempt A-10 face the same issue: there is no authoritative reference implementation of lossy DCOPF against which to validate results. Each tool implements loss approximations differently (iterative loss injection, piecewise linear loss curves, etc.), and without a common reference, cross-tool comparison is meaningless.

**Proposed Remedy:** Replace MATPOWER reference validation with internal consistency checks: verify that total losses are positive, that loss allocation is monotonic with branch flow, and that the lossy objective exceeds the lossless objective.

</details>

### T10: ACPF at 10k-Bus Is a Genuine Discriminator {#t10}

<details className="eval-details">
<summary>
T10: ACPF at 10k-Bus Is a Genuine Discriminator
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#f3e5f5', color: '#6a1b9a'}}>Tool-Specific</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffcdd2', color: '#b71c1c'}}>HIGH</span>
</summary>

**Affected Tools:** 6 | **Evidence:** 6 findings

A-2 MEDIUM (AC power flow on 10,000-bus ACTIVSg10k) produces three distinct outcome classes with strong discriminative value, making it one of the most informative tests in the entire evaluation. The three classes are: (1) converges with diagnostics -- pandapower, GridCal, and MATPOWER all converge using DC warm starts and report convergence error metrics; (2) fails with minimal diagnostics -- PowerModels and PowerSimulations fail without providing iteration counts, residuals, or configurable tolerances; and (3) false convergence -- PyPSA reports zero Newton-Raphson iterations with 83% of buses at flat start.

Probe-001 demonstrated that PyPSA's `n.pf()` performs zero NR iterations on ACTIVSg10k, returning `converged: False` while 8,274 of 10,000 buses retain their flat-start voltage of exactly 1.0 pu. Probe-024 confirmed that PowerSimulations returns `Missing` in 5.5s with zero diagnostic information. These are qualitatively different failure modes that reveal meaningful differences in solver robustness and diagnostic quality.

**Proposed Remedy:** Retain A-2 MEDIUM as-is; it is working as intended. Add diagnostic quality sub-scoring to capture the difference between informative failure and silent failure.

**Related Probes:** [probe-001](./probe-results#probe-001)

</details>

### T11: SCOPF and N-M Sweep Infeasibility at MEDIUM Scale {#t11}

<details className="eval-details">
<summary>
T11: SCOPF and N-M Sweep Infeasibility at MEDIUM Scale
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#fce4ec', color: '#c62828'}}>Protocol Gap</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffe0b2', color: '#e65100'}}>MEDIUM</span>
</summary>

**Affected Tools:** 6 | **Evidence:** 3 findings

C-8 MEDIUM (500-contingency SCOPF on ACTIVSg10k) is a unanimous fail across all six tools. The N-M sweep at x=5 branches, m=4 simultaneous failures on MEDIUM is infeasible at N-3 and above for all tools. Even PowerModels -- which handles SCOPF natively -- times out at 567s with just 5 contingencies on MEDIUM (probe-016). The combinatorial explosion at higher contingency levels (C(1000,3) = 166 million combinations) makes this structurally intractable.

Since all tools fail identically, C-8 MEDIUM provides zero discriminative signal. The useful information is at N-1 and N-2 contingency levels, where tools show meaningful variation in both capability and performance. The MEDIUM-scale SCOPF test is effectively a ceiling test that confirms current solver technology limits rather than differentiating tool implementations.

**Proposed Remedy:** Reduce C-8 MEDIUM contingency count to a level where at least one tool can complete, or re-scope as a timeout-tolerance test measuring how gracefully tools handle infeasible problems.

</details>

### T12: Solver Capability vs Tool Capability Conflation {#t12}

<details className="eval-details">
<summary>
T12: Solver Capability vs Tool Capability Conflation
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#fff3e0', color: '#e65100'}}>Scoring Inconsistency</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#ffe0b2', color: '#e65100'}}>MEDIUM</span>
</summary>

**Affected Tools:** 4 | **Evidence:** 4 findings

Multiple tests conflate open-source solver limitations with tool capability. The protocol mandates single-threaded open-source solvers (HiGHS for LP/MIP, Ipopt for NLP), creating a performance ceiling that is independent of the tool's own architecture. A tool that elegantly formulates a problem but relies on HiGHS for the solve step will time out at the same scale as a tool with a crude formulation using the same solver.

A-5/C-4 (unit commitment) is the clearest example: HiGHS single-threaded MIP timeout on 2000-bus UC affects both PyPSA and PowerModels equally. C-7 (solver swap) is graded as `qualified_pass` for tools that can swap solvers via a parameter change, but the test's discriminative value is limited because the solver failure is what triggered the swap in the first place.

**Proposed Remedy:** Separate formulation expressiveness (can the tool express the problem?) from solver scalability (can the solver solve it at scale?). Report solver-bounded failures distinctly from formulation failures.

</details>

### T13: Maturity Metrics Based on Research Without Verification {#t13}

<details className="eval-details">
<summary>
T13: Maturity Metrics Based on Research Without Verification
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#e8f5e9', color: '#2e7d32'}}>Data Quality</span>
{' '}<span style={{padding: '2px 8px', borderRadius: '4px', fontSize: '0.85em', backgroundColor: '#c8e6c9', color: '#2e7d32'}}>LOW</span>
</summary>

**Affected Tools:** 6 | **Evidence:** 6 findings

All E-series (maturity) metrics are based on desk research -- reading documentation, README badges, GitHub statistics, and published papers -- without independent verification. Probe-025 demonstrated the risk: PowerSimulations' E-6 claimed 100% code coverage based on a Codecov badge, but the actual badge displays 78% coverage. The 22 percentage-point discrepancy was not a rounding error but a misread of cached or outdated badge rendering.

Other maturity numbers -- commit counts, contributor percentages, deployment claims, release cadence -- are plausible but similarly unverified. For a non-scored informational dimension, this is acceptable. But if maturity metrics were to influence tool selection decisions, the current evidence standard is insufficient. The risk is that well-marketed tools with polished READMEs score higher than tools with better actual practices but less visible metrics.

**Proposed Remedy:** Require verification of quantitative claims: check Codecov/Coveralls detail pages (not just badges), verify contributor counts against the GitHub API, and cross-reference release cadence with actual tag dates.

**Related Probes:** [probe-025](./probe-results#probe-025)

</details>

## Tool-Specific Notes

### gridcal-F02: TapPhaseControl bug cascade {#gridcal-f02}

GridCal's TapPhaseControl issue blocks multiple downstream evaluations. It is the immediate cause of the reconstructed A-5 and A-6 failures and also cascades into stochastic/scalability outcomes that were reported separately in criterion pages.

### gridcal-F04: PTDF mismatch on MEDIUM {#gridcal-f04}

GridCal reproduces the same ACTIVSg10k PTDF discrepancy seen in pandapower: up to 743 MW branch-flow error when the standard PTDF formula is used without phase-shifter correction terms. This note is the GridCal-specific instance of [T06](#t06).

### gridcal-F05: MEDIUM DCOPF has no binding branches {#gridcal-f05}

GridCal's MEDIUM-scale DCOPF result is dominated by network insufficiency rather than solver behavior: branch loadings remain below binding limits, so the pass reflects formulation completion on an uncongested case rather than economically discriminative OPF behavior. This note is the GridCal-specific instance of [T01](#t01).

### psi-F02: PowerSimulations unit mismatch risk {#psi-f02}

PowerSimulations exposes dispatch from `read_variables()` in MW while related limit accessors are commonly interpreted in system-base pu. The underlying optimization result is valid, but the mixed unit conventions create a transfer risk in downstream analysis and explain the criterion-page caution around AC feasibility interpretation.

### psi-F03: Dual extraction verified only for non-binding case {#psi-f03}

The extensibility sweep confirmed that PowerSimulations users can reach JuMP duals, but the original criterion evidence only demonstrated the non-binding case. That leaves binding-constraint dual behavior under-documented in the report narrative even though the API path itself is viable.

### psi-F05: SCUC on case39 shows no cycling {#psi-f05}

PowerSimulations can formulate and solve SCUC natively, but IEEE 39-bus does not force startup or shutdown decisions. The resulting all-committed schedule weakens the discriminative value of the SCUC criterion and is the PowerSimulations-specific instance of [T03](#t03).

### matpower-F01: Distributed slack timing anomaly {#matpower-f01}

MATPOWER's distributed-slack MEDIUM run completes, but the observed wall-clock time is anomalously high relative to expectation for the same formulation family. The report attributes this to dense matrix handling in the manual `opt_model` construction rather than a capability failure.

### matpower-F02: Contingency runtime dominated by Octave overhead {#matpower-f02}

MATPOWER's C-5 runtime is dominated by Octave `containers.Map` bookkeeping rather than contingency-screening math. The underlying LODF screening step remains fast, so this finding is about implementation overhead, not the contingency method itself.

### matpower-F05: SCUC on case39 shows no cycling {#matpower-f05}

MATPOWER's MOST formulation passes SCUC structurally, but case39 keeps all units committed for the full horizon. This confirms capability while also showing that the reference network does not exercise actual commitment switching behavior, making this the MATPOWER-specific instance of [T03](#t03).
