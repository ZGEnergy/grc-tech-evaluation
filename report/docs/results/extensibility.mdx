# Extensibility

## Overview

Extensibility measures the ability to add custom constraints, swap solvers,
modify networks programmatically, and integrate with external tools and
workflows. It is a high-priority criterion, reflecting the requirement that a
selected tool must accommodate custom modeling needs (flow gate constraints,
solver benchmarking, PTDF-based analysis, and programmatic network
modification) without requiring source code patches or fragile internal API
access.

Tests B-1 through B-9 map to this criterion: custom constraint injection with
dual extraction (B-1), graph and topology access (B-2), contingency loop
efficiency (B-3), stochastic wrapping (B-4), interoperability and export
(B-5), code architecture assessment (B-6), AC feasibility extension (B-7),
reference bus configuration (B-8), and PTDF extraction with phase-shifter
correction (B-9).

| Tool | Tier | Rationale |

| **PowerModels.jl** | **Strong** | Two-level API enables custom constraint injection in 4 LOC with dual extraction (B-1). First-class PTDF with machine-epsilon accuracy at 10,000 buses (B-9). Four-layer dispatch architecture (B-6). |

## Per-Tool Analysis

### PyPSA (Strong)

PyPSA achieves a clean sweep across all extensibility tests. The
`extra_functionality` callback provides a documented hook for custom constraint
injection with full access to the linopy model, requiring only 2 lines of
code for a flow gate constraint with dual extraction (B-1). NetworkX graph
access is zero-friction via `n.graph()` (B-2, 4 LOC for BFS and subgraph
extraction). PTDF extraction is native via `SubNetwork.calculate_PTDF()` with
machine-precision flow agreement (max error 1.9e-14 p.u. at TINY, B-9).

**Key strengths:**

- Custom constraint injection via `extra_functionality` callback with linopy model access; 2 LOC for flow gate constraint with dual extraction (B-1).
- Native NetworkX graph export via `n.graph()` for immediate access to BFS, DFS, and shortest-path algorithms (B-2).
- N-M contingency sweep via `n.copy()` plus `n.lpf()` with no file re-reads; 3,276 cases in 213s (B-3).
- Programmatic time series injection via DataFrame assignment for stochastic scenarios (B-4).
- DataFrame-native results with zero-friction export to CSV, Parquet, and HDF5 (B-5).
- Well-separated 4-layer architecture with 5 documented injection points (B-6).
- Native PTDF extraction with correct phase-shifter handling (B-9).

**Key weaknesses:**

- `n.lpf_contingency()` broken on Python 3.12+; does not block N-M contingency analysis (BODF or copy-plus-lpf patterns work) (B-3).
- DCPF solver (scipy.sparse) is hardcoded and not swappable via parameter (B-6).

**Workarounds:**

None required across all extensibility tests. All functionality achieved through documented public APIs.

### PowerModels.jl (Strong)

PowerModels provides the cleanest constraint injection pattern among all
evaluated tools via its two-level API (`instantiate_model`/`optimize_model!`),
which exposes the JuMP model for arbitrary modification. Custom flow gate
constraints with dual extraction work at both TINY and MEDIUM scale (B-1).
The native PTDF pipeline via `calc_basic_ptdf_matrix` achieves flow prediction
error below 1e-11 at 10,000 buses (B-9). Contingency loops are efficient
via dict `deepcopy` plus `br_status=0` (B-3, 50 N-1 cases at 1.25ms per case
on TINY).

**Key strengths:**

- Two-level API provides custom constraint injection in 4 LOC with dual extraction (B-1).
- First-class PTDF computation via `calc_basic_ptdf_matrix` with machine-epsilon accuracy at MEDIUM scale (B-9).
- Four-layer dispatch architecture (public API, model lifecycle, formulation build, solver) with clean separation of concerns (B-6).
- N-M contingency sweep via `deepcopy` plus `br_status=0` at 1.25ms per case (B-3).
- Stochastic wrapping via `deepcopy` plus dict mutation at 4ms per scenario (B-4).
- Interoperability: 3 LOC per component type to export to DataFrame and CSV (B-5).

**Key weaknesses:**

- No Graphs.jl integration; adjacency must be built manually from branch `f_bus`/`t_bus` pairs (approximately 12 LOC, B-2 qualified pass).
- Formulation-specific methods in `form/*.jl` lack docstrings (B-6).

**Workarounds:**

- B-2 graph access: Manual adjacency construction from `data["branch"]`, approximately 12 LOC (durability: stable).
- B-8 distributed slack: Requires approximately 150 LOC custom PTDF-based JuMP OPF (durability: stable, but high effort).

### PowerSimulations.jl (Strong)

PowerSimulations inherits JuMP model access from its PowerModels foundation,
enabling custom constraint injection via standard JuMP `@constraint` macros
(B-1). The 5-layer package architecture (PowerSystems for data,
PowerNetworkMatrices for PTDF/LODF, PowerFlows for power flow,
PowerSimulations for optimization, results via DataFrames) provides the
cleanest separation of concerns among all evaluated tools (B-6).
Machine-precision PTDF verification with max error of 1.15e-14 p.u.
is achieved via a single-call API (B-9).

**Key strengths:**

- Full JuMP model access via `PSI.get_jump_model()` enables arbitrary constraint addition and dual extraction (B-1).
- Machine-precision PTDF computation via single-call API; max error 1.15e-14 p.u. (B-9).
- Clean 5-layer package architecture with DAG dependency structure and no circular dependencies (B-6).
- DataFrame-native results enable 2-LOC CSV export (B-5).
- Reference bus configurable via `set_bustype!()` API with LMP invariance confirmed to machine precision (B-8).
- 20-scenario stochastic wrapping completes in 2.1s total, 0.1s mean per scenario (B-4).

**Key weaknesses:**

- No Graphs.jl integration; BFS/DFS requires manual implementation over the adjacency matrix (B-2 qualified pass).
- LODF superposition is approximate for M>1 contingencies; no Woodbury correction available (B-3).
- Time series immutability forces full System reconstruction per scenario (B-4).

**Workarounds:**

- B-2 graph access: Manual BFS over `AdjacencyMatrix` for graph traversal (durability: stable).
- B-3 contingency sweep: LODF screening pipeline assembled from 3 separate packages (durability: stable).
- B-4 stochastic wrapping: Full System reconstruction per scenario due to time series immutability (durability: stable).

### pandapower (Adequate)

pandapower is highly extensible for its core domain (power flow, contingency,
topology) with clean APIs and efficient in-place modification. NetworkX
integration via `create_nxgraph()` with impedance-weighted edges is seamless
(B-2). In-place branch switching enables efficient contingency loops at
6.4ms per case on TINY (B-3). The pandas-native data model provides trivial
DataFrame export (B-5).

The critical extensibility gap is OPF constraint injection. The embedded
PYPOWER solver is a closed system with no public hook for custom constraints
or dual extraction. The only path requires monkey-patching
`_optimal_powerflow` and capturing the PYPOWER result dict (B-1 partial pass,
fragile workaround). This single limitation has cascading consequences for
SCOPF, interface flow limits, and any formulation beyond the built-in
constraint set.

**Key strengths:**

- First-class NetworkX graph bridge via `create_nxgraph()` with impedance weighting and switch awareness (B-2).
- Clean N-M contingency sweep via in-service toggling; 3,276 cases at 6.4ms each (B-3).
- DataFrame-native results with trivial CSV export in 4 LOC (B-5).
- Clean 6-layer architecture with well-separated concerns (B-6).
- PTDF computation via native `makePTDF()` with machine-precision accuracy on TINY (B-9).

**Key weaknesses:**

- No public API for custom OPF constraint injection; monkey-patching `_optimal_powerflow` required (B-1 partial pass, fragile).
- OPF result dict (containing duals and multipliers) discarded during result extraction; users needing constraint duals must access undocumented internals (B-1).

**Workarounds:**

- B-1 custom constraints: Replicate `_optimal_powerflow`, monkey-patch module reference, capture PYPOWER result dict (durability: fragile).
- B-8 reference bus: 5-6 API calls to remove/recreate `ext_grid` plus `gen` elements with manual cost transfer (durability: stable).

### GridCal (Adequate)

GridCal provides strong graph access and topology operations via its native
NetworkX integration, and its pure-Python architecture enables source-level
inspection. Contingency loops and interoperability pass at TINY scale. PTDF
and LODF matrices are accessible as dense NumPy arrays via one-line
`linear_power_flow()` (B-9). Slack bus reconfiguration via `bus.is_slack`
boolean toggle is clean, and LMPs are invariant to slack choice due to the
PTDF-based formulation (B-8). Solver swap is a single parameter change
(`mip_solver=MIPSolvers.X`) with no reformulation needed (B-4).

The principal gap is identical to pandapower: no public API for custom
constraint injection into the OPF. The monolithic OPF formulation
(`linear_opf_ts.py`, 3,146 LOC as a single procedural function) provides no
extension points or plugin architecture.

**Key strengths:**

- NetworkX graph via `build_graph()` enables BFS, shortest-path, and subgraph extraction (B-2).
- Branch toggle via `branch.active` for contingency sweeps with no model reconstruction (B-3).
- PTDF and LODF matrices as dense NumPy arrays via one-line API (B-9).
- Parameter-only solver swap via PuLP (`mip_solver=MIPSolvers.X`) with no reformulation (B-4).

**Key weaknesses:**

- No public API for custom constraint injection (B-1 partial pass).
- Monkey-patch required to add flowgate constraints; relies on internal naming convention `br_flow_upper_lim_0_<idx>` (B-1 fragile).
- `TapPhaseControl` enum bug crashes time-series OPF on networks with transformers in the evaluated version (B-4).
- Monolithic OPF formulation (3,146 LOC single file) with no hook points or plugin architecture (B-6).

**Workarounds:**

- B-1 custom constraints: PulpLpModel.solve monkey-patch with internal naming conventions (durability: fragile).
- B-4 stochastic wrapping: Sequential snapshot OPF instead of time-series driver to avoid the `TapPhaseControl` bug; loses inter-temporal coupling (durability: stable).

### MATPOWER (Strong, Reference Only)

:::note[Reference Implementation]
MATPOWER served as the reference implementation for validating test protocols
and network data. It is not a candidate for Phase 2 selection because the
customer requires inspectable source code, which precludes the compiled
MATLAB runtime. Grades reflect protocol validation performance only.
:::

MATPOWER provides broad extensibility through its documented `mpc.A/l/u`
framework for custom constraint injection with dual value extraction at
`results.lin.mu.u.usr` (B-1). Native `makePTDF()`/`makeLODF()` enable
efficient LODF-based contingency screening; 126 contingencies processed in
12.1s (B-3). Single-call PTDF computation achieves machine-precision flow
accuracy with max error of 1.5e-12 MW (B-9). Reference bus reconfiguration
is a single matrix element assignment, and LMPs are invariant to slack
choice (B-8).

**Key strengths:**

- Documented custom constraint API (`mpc.A/l/u`) with dual value extraction (B-1).
- Native `makePTDF()`/`makeLODF()` with machine-precision accuracy (B-9).
- Flat `mpc` struct data model enables zero-friction programmatic network modification (B-8).
- 12+ solver backends accessible via a single `mpoption` parameter (B-6).

**Key weaknesses:**

- MIPS solver fails with user constraints on QP problems due to numerical singularity (B-1).
- No DataFrame abstraction; CSV export requires manual `fopen`/`fprintf`/`fclose` pattern (B-5 qualified pass).

**Workarounds:**

- B-1 custom constraints: GLPK (LP) with PWL-linearized costs instead of MIPS (QP) (durability: stable).
- B-5 CSV export: Manual header writing via standard Octave I/O (durability: stable).

## Summary

Extensibility separates tools into two clear tiers. PyPSA, PowerModels, and
PowerSimulations all earn Strong, each providing a clean, documented path for
custom constraint injection with dual extraction, machine-precision PTDF
computation, and architectures designed for extension. PyPSA's
`extra_functionality` callback, PowerModels' two-level API, and
PowerSimulations' JuMP model access represent three distinct but equally
effective extension mechanisms. MATPOWER (reference only) also earns Strong
via its documented `mpc.A/l/u` framework.

pandapower and GridCal both earn Adequate. Both tools provide excellent graph
access and efficient contingency sweep infrastructure, but both share the same
critical gap: no public API for custom OPF constraint injection. In both
tools, the only path to custom constraints requires monkey-patching internal
solver objects (fragile workaround). This limitation directly constrains
Phase 2 workflows that require flow gate constraints or custom security
formulations.

The key discriminator across all tools is the custom constraint API (B-1).
Tools with documented public extension APIs (PyPSA, PowerModels, MATPOWER)
enable constraint injection in 2-4 LOC. Tools without such APIs (pandapower,
GridCal) require 200+ LOC fragile workarounds that depend on internal naming
conventions and could break on any version upgrade.
