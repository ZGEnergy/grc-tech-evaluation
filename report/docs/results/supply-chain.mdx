# Supply Chain

## Overview

Supply Chain, Inspectability, and Licensing Risk is the **gate criterion**. A tier of Weak or Failing is disqualifying regardless of performance on all other criteria. The Suite F tests assess core licensing (F-1), dependency tree transparency (F-2), dependency license audit (F-3), compiled extension audit (F-4), code inspectability (F-5), distribution integrity (F-6), air-gap installability (F-7), solver dependency analysis (F-8), and getting-started artifact integrity (F-9). These tests ensure that a selected tool can be deployed in the customer's operational environment without licensing conflicts, uninspectable binaries, or supply chain vulnerabilities.

All six tools passed the gate threshold. This makes Supply Chain the least differentiating criterion; the primary distinctions are licensing model, dependency tree depth, and distribution integrity practices.

### PyPSA — <span className="tier-strong">Strong</span>

</summary>

PyPSA has a clean supply chain profile. The MIT license is fully permissive with no copyleft obligations (F-1). The dependency tree consists of 17 direct packages (approximately 70 transitive) rooted in standard Python scientific computing (numpy, scipy, pandas, linopy) with a maximum depth of 4 (F-2). One GPL-2.0 direct dependency (Levenshtein, used for fuzzy attribute name matching) is flagged; it is replaceable with the MIT-licensed `rapidfuzz`, which is already a transitive dependency (F-3). All 249 `.so` files in dependencies have publicly available source (F-4). The full execution path from API call to solver invocation is readable Python with zero opaque steps (F-5). Distribution is via PyPI with SHA-256 hashes and automated CI/CD release; no Sigstore attestations (ecosystem-wide gap) (F-6). Air-gap installation is achievable by pre-downloading wheels with no runtime network dependencies (F-7). The bundled HiGHS solver (MIT) covers LP, MILP, and QP with no commercial solver requirement (F-8). Getting-started examples are self-contained with no mutable URLs (F-9).

### pandapower — <span className="tier-strong">Strong</span>

</summary>

pandapower has an exemplary supply chain profile. The BSD-3-Clause license is permissive with no copyleft obligations (F-1). The dependency tree is compact: 39 packages with a maximum depth of 3, all versions pinned via `uv.lock` (F-2). All dependencies carry permissive licenses except the optional LightSim2Grid accelerator (MPL-2.0, weak copyleft, file-level only), which is excludable (F-3). All 216 compiled extensions (`.so` files) have publicly available source (F-4). The entire power flow execution path is readable Python; the sole compiled step is scipy's SuperLU factorization (F-5). pandapower is the only evaluated tool with Sigstore provenance attestations and Trusted Publishing via GitHub Actions (F-6). Air-gap installation is achievable via pre-downloaded wheels with no runtime network access (F-7). The embedded PYPOWER solver eliminates any external solver dependency (F-8). Tutorials are not shipped inside the pip package (must clone the repository) and official documentation uses an unversioned `pip install pandapower` command (F-9).

### GridCal — <span className="tier-strong">Strong</span>

</summary>

:::caution[MPL-2.0 License]
GridCal uses the Mozilla Public License 2.0, which requires that modifications to GridCal source files be distributed under MPL-2.0. This is a file-level copyleft; it does not affect code that merely imports GridCal, but any patches to GridCal itself must be shared under MPL-2.0.
:::

GridCal (VeraGridEngine) is 100% pure Python (`py3-none-any` wheel) with no compiled extensions in the core package (F-4). The MPL-2.0 license changed from LGPL in November 2024 and is compatible with enterprise adoption (F-1). The dependency tree includes 62 packages (28 direct) with 2 LGPL packages (chardet, moocore), both low-risk; no GPL, AGPL, or proprietary licenses were found (F-2, F-3). The full DCPF execution path is traceable through 4 layers of Python in approximately 200 lines, with a custom interior-point solver for AC OPF that eliminates any Ipopt dependency (F-5). Distribution is sdist-only on PyPI with no release signing; SHA-256 hashes are available (F-6). Air-gap installation is feasible; all compiled dependencies have platform wheels on PyPI, producing an approximately 500-700 MB bundle (F-7). HiGHS (MIT) is bundled via PuLP, and CBC is also bundled; no proprietary solver is required (F-8). The single-maintainer publishing model (no Trusted Publisher, no signing) concentrates supply chain risk on one PyPI account (F-6).

Air-gap installation is feasible via Julia depot caching with `JULIA_PKG_OFFLINE=true` (F-7). Getting-started examples use `/stable/` documentation links but include mutable badge references (F-9).

## Summary

All six tools pass the supply chain gate. The Python tools (PyPSA, pandapower, GridCal) and MATPOWER earn Strong, reflecting permissive or file-level-copyleft licenses, fully inspectable source, and straightforward air-gap installation (F-1, F-5, F-7). The Julia tools (PowerModels, PowerSimulations) earn Adequate due to larger dependency trees (114 and 184 packages respectively), optional GPL-licensed solver wrappers that require explicit exclusion, and mutable getting-started artifacts (F-2, F-3, F-9). pandapower's Sigstore provenance attestations (F-6) and MATPOWER's zero-dependency self-contained distribution (F-2) are best-in-class practices within their respective tiers.

The most common supply chain risk across all tools is the optional presence of GPL-licensed GLPK solver bindings. For any tool selected for Phase 2, the deployment configuration should explicitly exclude GLPK and rely on HiGHS (MIT) or Ipopt (EPL 2.0) for all optimization workloads.
