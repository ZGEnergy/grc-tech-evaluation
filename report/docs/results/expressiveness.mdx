# Expressiveness

## Overview

Expressiveness measures the breadth of power system analyses a tool can perform
natively or with minimal extension. It is the highest-priority criterion in
the rubric (weight 0.30), reflecting the core requirement that a selected tool
must cover the target analysis types without rewriting fundamental algorithms.
Tests A-1 through A-12 map to this criterion: DC power flow (A-1), AC power
flow (A-2), DC OPF with LMP extraction (A-3), AC feasibility checking (A-4),
security-constrained unit commitment (A-5), economic dispatch (A-6),
contingency sweeps (A-7), stochastic optimization (A-8), SCOPF (A-9), lossy
DC OPF with LMP decomposition (A-10), distributed slack OPF (A-11), and
multi-period DC OPF with storage (A-12).

Each tool was tested at three network scales (TINY at 39 buses, SMALL at
2,000 buses, and MEDIUM at 10,000 buses) to expose both capability gaps and
scale-dependent failures. Tools that pass at TINY but fail at SMALL or MEDIUM
reveal API limitations that would surface in production use.

|------|------|-----------|
| **PyPSA** | **Strong** | 8 of 10 tests pass natively, including SCOPF (A-9), lossy DC OPF (A-10), and multi-period storage (A-12). One blocking gap: distributed slack OPF (A-11). |
| **pandapower** | **Weak** | 4 of 10 tests pass; 4 independent blocking failures in SCUC (A-5), lossy OPF (A-10), distributed slack (A-11), and multi-period storage (A-12). Design scope limited to single-period steady-state analysis. |
| **GridCal** | **Adequate** | Covers core PF/OPF and SCOPF (A-9) via native LODF formulation. Two blocking formulation bugs: battery energy balance sign error (A-12) and distributed slack hardcoded off (A-11). Soft branch constraints in DC OPF (A-3) reduce formulation fidelity. |

## Per-Tool Analysis

### PyPSA (Strong)

PyPSA demonstrates the broadest native expressiveness among all evaluated
tools. It passes A-1 (DCPF), A-2 (ACPF), A-3 (DC OPF), and A-4 (AC
feasibility) cleanly at TINY scale with correct convergence diagnostics and
structured output. Built-in SCUC via committable generators with min up/down
times and startup costs passes at TINY (A-5). SCOPF via the native
`optimize_security_constrained()` method passes at both TINY and SMALL scale
(A-9), making PyPSA one of two tools with a clean SCOPF path. Native lossy
DC OPF with piecewise-linear loss approximation passes at TINY (A-10), and
multi-period DC OPF with storage passes with correct BESS arbitrage behavior
(A-12).

**Key strengths:**

- Native SCOPF via BODF-based N-1 contingency constraints embedded in the LP formulation (A-9).
- Lossy DC OPF via `transmission_losses` parameter with piecewise-linear loss approximation (A-10).
- Multi-period storage OPF with cyclic state-of-charge constraints (A-12).
- Built-in SCUC with all required constraint types as native attributes (A-5).

**Key weaknesses:**

- Distributed slack DC OPF is architecturally blocked; the linopy model has no bus angle variables, making distributed slack formulation impossible in the OPF context (A-11). AC PF distributed slack works via `n.pf(distribute_slack=True)`.
- Branch shadow prices (`mu_upper`/`mu_lower`) not populated after `n.optimize()` in the evaluated version; extraction requires accessing linopy constraint duals directly (A-3, A-12).

**Workarounds:**

- A-6 SCED: Manual `p_min_pu`/`p_max_pu` bound construction to fix the commitment schedule (durability: stable).
- A-12 shadow prices: Extract from linopy constraint duals instead of the PyPSA accessor (durability: fragile).
- A-11 distributed slack: Available in PF context but blocked in OPF context (durability: blocking).

### pandapower (Weak)

pandapower has a well-designed API for power flow analysis. A-1 (DCPF),
A-2 (ACPF), A-3 (DC OPF), and A-4 (AC feasibility) all pass cleanly at
TINY scale with structured DataFrame output. The contingency sweep (A-7)
passes at all scales using efficient in-place branch switching. However,
the Weak tier reflects fundamental architectural limitations: four
independent blocking failures prevent the tool from expressing core
market-operations problems.

**Key strengths:**

- Clean DCPF, ACPF, DC OPF, and AC feasibility workflow within a single network object (A-1 through A-4).
- Efficient N-M contingency sweep via in-service toggling (A-7).
- Embedded PYPOWER solver requires no external solver installation.

**Key weaknesses:**

- No MILP optimization capability; no binary commitment variables (A-5 fail).
- No lossy DC OPF and no LMP decomposition into energy, congestion, and loss components (A-10 fail).
- Distributed slack silently ignored in OPF; `rundcopp(distributed_slack=True)` is absorbed by `**kwargs` with no warning (A-11 fail).
- No multi-period DC OPF with inter-temporal storage coupling (A-12 fail).
- No contingency-constrained optimization; SCOPF requires bypassing pandapower's OPF entirely to build a manual LP (A-9).

**Workarounds:**

- A-6 SCED: Sequential single-period DC OPF with manual ramp-rate bound manipulation (durability: fragile).
- A-9 SCOPF: Manual PTDF/LODF construction plus `scipy.optimize.linprog`, bypassing pandapower OPF entirely (durability: blocking).

:::warning
pandapower's four independent expressiveness failures (A-5, A-10, A-11, A-12)
are architectural boundaries, not missing features. The tool's design scope
is limited to single-period steady-state analysis. These gaps cannot be
resolved through configuration or extension.
:::

### GridCal (Adequate)

GridCal provides clean power flow and DC OPF capability with a 3-line API.
A-1 through A-4 pass at TINY scale, and the native LODF-based SCOPF via
`consider_contingencies=True` (A-9) is a notable strength, providing a joint
formulation rather than iterative Benders decomposition. Native 24-hour SCUC
with `OpfDispatchMode.UnitCommitment` is present (A-5 qualified pass), and
lossy DC OPF exists via an `add_losses_approximation` flag (A-10
constrained pass), though the loss approximation underestimates by over 500x.

**Key strengths:**

- Native LODF-based SCOPF via `consider_contingencies=True` with joint formulation (A-9).
- Native 24-hour SCUC with ramp rates and min up/down constraints (A-5).
- Clean 3-line power flow API with first-class convergence diagnostics (A-2).

**Key weaknesses:**

- Battery energy balance sign error in multi-period OPF: energy increases during discharge, producing physically impossible results (A-12 fail, blocking).
- Distributed slack hardcoded off in OPF formulation (`distributed_slack=False` at source level); silently ignored when set to True (A-11 fail, blocking).
- Soft branch flow constraints in DC OPF allow overloading (112% observed on TINY); no hard-constraint mode available (A-3 partial pass).
- Loss approximation underestimates by over 500x (A-10); formula uses static thermal rating instead of actual flow.

**Workarounds:**

- A-6 SCED: Pmax/Pmin profile zeroing for decommitted generators (durability: stable).
- A-11 distributed slack: No workaround available (durability: blocking).
- A-12 multi-period storage: No workaround available (durability: blocking).

### PowerModels.jl (Adequate)

PowerModels provides mathematically rigorous OPF formulations with native
support for multiple AC OPF relaxations (SOC, QC, SDP) and a clean
JuMP-based architecture. Core PF/OPF tests (A-1 through A-4) pass at TINY
scale. The lossy DC OPF via `DCPLLPowerModel` with three-component LMP
decomposition (energy, congestion, loss) is a key differentiator (A-10
qualified pass). Multi-period OPF with storage via `replicate()` and binary
charge/discharge complementarity passes at TINY (A-12).

**Key strengths:**

- Lossy DC OPF with full three-component LMP decomposition (A-10).
- Multi-period storage OPF with cyclic state-of-charge constraint (A-12).
- SCOPF mechanism via PTDF/LODF and iterative Benders cutting plane through the two-level API (A-9 qualified pass).
- Clean one-call DC OPF with native LMP extraction via `lam_kcl_r` dual variable (A-3).

**Key weaknesses:**

- No native SCUC; unit commitment binary variables, min up/down times, and startup costs are entirely absent (A-5 fail).
- No distributed slack formulation; all 18 built-in formulations use a single reference bus (A-11 fail).
- `compute_ac_pf` does not expose NR iteration count or convergence residual (A-2 qualified pass).

**Workarounds:**

- A-9 SCOPF: Manual Benders cutting plane with LODF injection (durability: stable).
- A-12 storage: Cyclic SoC constraint via manual JuMP injection; SCIP required (durability: stable).
- A-11 distributed slack: Requires approximately 150 LOC custom PTDF-based JuMP OPF (durability: blocking).
- A-6 SCED: Multi-period `DCPPowerModel` with ramp constraints via `replicate()` (durability: stable).

### PowerSimulations.jl (Adequate)

PowerSimulations is the only evaluated tool with native SCUC/SCED support
via built-in `ThermalStandardUnitCommitment` formulations. A-5 (SCUC) and
A-6 (SCED) achieve qualified pass at TINY, though both require an
`initialize_model=false` workaround due to initialization bugs. SCOPF is
achievable via manual LODF-based JuMP constraint assembly (A-9 qualified
pass).

**Key strengths:**

- Built-in unit commitment formulations with binary commitment, ramp, and min up/down constraints (A-5).
- Native DC and AC power flow via PowerFlows.jl with sub-millisecond solve times (A-1, A-2).
- SCOPF achievable via LODF-based manual constraint assembly with 17.7% cost increase over unconstrained OPF (A-9).

**Key weaknesses:**

- Lossy DC OPF blocked by solver-formulation mismatch; `DCPLLPowerModel` uses quadratic constraints unsupported by HiGHS, GLPK, and SCIP (A-10 fail, blocking).
- No distributed slack formulation exists in PowerSimulations or PowerModels (A-11 fail, blocking).
- No built-in storage formulations; `EnergyReservoirStorage` exists as a data type but PowerSimulations has no `construct_device!` method for it (A-12 qualified pass via fragile manual JuMP injection).
- No built-in UC-to-ED handoff; commitment transfer requires manual JuMP variable fixing (A-6).

**Workarounds:**

- A-5 SCUC: `initialize_model=false` plus `JuMP.optimize!()` to bypass broken initialization (durability: fragile).
- A-12 storage: Manual JuMP BESS variables and constraints depending on PSI internal constraint naming (durability: fragile).
- A-9 SCOPF: Manual LODF plus JuMP constraint injection (durability: stable).

### MATPOWER (Adequate, Reference Only)

:::note[Reference Implementation]
MATPOWER served as the reference implementation for validating test protocols
and network data. It is not a candidate for Phase 2 selection because the
customer requires inspectable source code, which precludes the compiled
MATLAB runtime. Grades reflect protocol validation performance only.
:::

MATPOWER provides broad native expressiveness for core power flow and OPF
via the legacy API, and MOST provides built-in SCUC, multi-period scheduling,
and storage capability. A-1 through A-4 pass cleanly. A-5 (SCUC) achieves
qualified pass; the MOST formulation is correct but a GLPK exit flag
integration bug prevents result extraction on the primary test case. A-12
(multi-period storage) passes with correct BESS arbitrage behavior under
linear cost constraints.

**Key strengths:**

- MOST provides built-in SCUC with all constraint types: min up/down, startup costs, ramp rates, reserves, storage (A-5).
- Native multi-period DC OPF with storage via `addstorage()` API (A-12).
- Clean two-stage UC/ED separability via `GEN_STATUS` toggling (A-6).

**Key weaknesses:**

- No native SCOPF function; manual LODF computation and user constraint injection required. Benders iteration encounters MIPS numerical issues (A-9 fail, blocking).
- No internal loss model in DC OPF; iterative loss injection produces correct total losses but zero marginal loss LMP components (A-10 fail, blocking).
- GLPK exit flag integration bug prevents SCUC result extraction for non-trivial problems (A-5).

**Workarounds:**

- A-5 SCUC: GLPK MIP gap exit code mapped to failure; validated on bundled test case (durability: stable).
- A-11 distributed slack: Post-processing via `makePTDF` with documented slack distribution vector (durability: stable).
- A-9 SCOPF: Benders iteration with LODF constraint injection via `mpc.A/l/u` (durability: blocking).
- A-10 lossy DC OPF: Iterative loss injection external to OPF; cannot produce non-zero loss LMP components (durability: blocking).

## Summary

PyPSA is the clear leader in expressiveness, earning a Strong tier with 8 of
10 tests passing natively and the broadest coverage of market-operations
formulations (SCOPF, lossy OPF, multi-period storage). Its sole blocking gap,
distributed slack OPF (A-11), is shared by every evaluated tool except
MATPOWER (where it is achievable via post-processing).

The Adequate tier contains four tools with distinct profiles. PowerModels
excels at OPF formulation rigor (lossy DC OPF with LMP decomposition) but
lacks unit commitment entirely. PowerSimulations uniquely provides native
SCUC but is blocked on lossy OPF and requires fragile workarounds for
several core workflows. GridCal covers a broad spectrum including native
SCOPF but is undermined by two blocking formulation bugs. MATPOWER has the
broadest built-in capabilities via MOST but is constrained by its Octave
solver ecosystem.

pandapower falls to Weak due to four independent blocking failures that
reflect fundamental architectural boundaries; the tool's design scope does
not extend beyond single-period steady-state analysis.
