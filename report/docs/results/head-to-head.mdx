# Head-to-Head Comparison

This page compares six capabilities critical to Phase 2 development across the five
candidate tools and the MATPOWER reference implementation. Each capability is rated
as **Native** (built-in support with no additional code), **Extension** (achievable
via the tool's public API with user-written code), **Workaround** (achievable only
through fragile internal API access or monkey-patching), or **Gap** (not feasible
within the tool's architecture). Ratings reflect observed test outcomes from the
A-series, B-series, C-series, and P2-series tests.

MATPOWER is included for reference but excluded from the primary ranking because the
customer requires inspectable source code, which precludes the compiled MATLAB runtime.

## Summary Table

| Capability | PyPSA | PowerModels | PowerSim. | pandapower | GridCal | MATPOWER* |
|-----------|-------|-------------|-----------|------------|---------|-----------|
| SCOPF | **Native** | Extension | Extension | Gap | **Native** | Workaround |
| Distributed Slack OPF | Workaround | Workaround | **Native** | Gap | Gap | Extension |
| PWL Cost Curves | Workaround | **Native** | **Native** | **Native** | Gap | **Native** |
| PSS/E RAW Parsing | Gap | Workaround | Workaround | Gap | Workaround | **Native** |
| Custom Constraints | **Native** | **Native** | **Native** | Workaround | Workaround | **Native** |
| UC/ED Pipeline | **Native** | Gap | **Native** | Gap | **Native** | **Native** |

## Capability Details

<details className="eval-details" open>
<summary>Security-Constrained OPF (SCOPF)</summary>

SCOPF is required for N-1 security analysis in production market operations. A tool
with native SCOPF avoids the need to build a custom contingency screening layer,
reducing Phase 2 development effort significantly.

| Tool | Rating | Evidence |
|------|--------|---------|
| PyPSA | **Native** | Native SCOPF via `extra_functionality` with LODF-based contingency constraints; 50 contingencies on 10,000-bus network solved in 30s solver time (A-9, C-8). |
| PowerModels | Extension | Manual Benders cutting plane with LODF injection via the two-level JuMP API; functional but convergence not demonstrated on feasible 10,000-bus instance within the time budget (A-9, C-8). |
| PowerSimulations | Extension | Manual LODF constraint injection into the JuMP model; HiGHS crashes with `OTHER_ERROR` on the 10,000-bus network (A-9, C-8). |
| GridCal | **Native** | Native LODF-based SCOPF via `consider_contingencies` parameter; 50 contingencies on 10,000-bus network solved in 32s with 956 MW aggregate redispatch (A-9, C-8). |
| pandapower | Gap | No SCOPF capability. Manual PTDF/LODF construction plus an external solver bypasses pandapower's OPF entirely; OOM at 10,000-bus scale (A-9, C-8). |
| MATPOWER* | Workaround | LODF constraint injection via `mpc.A/l/u` user constraint API; N-1 infeasible on test case with MIPS numerical issues; blocking durability (A-9). |

</details>

<details className="eval-details">
<summary>Distributed Slack OPF</summary>

Distributed slack modeling is essential for realistic loss allocation and LMP
decomposition in large multi-area studies. A single-slack approximation introduces
systematic bias in flow patterns and congestion pricing that compounds with network
size.

| Tool | Rating | Evidence |
|------|--------|---------|
| PyPSA | Workaround | Available in power flow context (`n.pf(distribute_slack=True)`) but blocked in OPF; PyPSA's flow-based OPF formulation lacks angle variables, making distributed slack architecturally impossible in OPF (A-11). |
| PowerModels | Workaround | Requires approximately 150 LOC of custom PTDF-based JuMP OPF code; no built-in distributed slack formulation exists across all 18 formulation types (A-11, B-8). |
| PowerSimulations | **Native** | Native distributed slack in power flow via PowerFlows.jl (A-11). |
| pandapower | Gap | No distributed slack formulation available in OPF; `distributed_slack=True` is silently ignored via `**kwargs` passthrough on `rundcopp()` (A-11). |
| GridCal | Gap | Distributed slack hardcoded off in OPF formulation (`distributed_slack=False` at source level); silently ignored when set to True (A-11). |
| MATPOWER* | Extension | Post-processing via `makePTDF(baseMVA, bus, branch, slack_weights)` with documented slack distribution vector; clean public API (A-11, B-9). |

</details>

<details className="eval-details">
<summary>Piecewise-Linear Cost Curves</summary>

Production market clearing commonly uses piecewise-linear bid curves. A tool without
native PWL support requires building the SOS2 formulation as an extension, adding
development effort and a potential source of modeling error in the cost-to-dispatch
mapping.

| Tool | Rating | Evidence |
|------|--------|---------|
| PyPSA | Workaround | Generator tranche decomposition approximates PWL; no native SOS2 support. The `extra_functionality` callback could support a custom SOS2 formulation but this has not been demonstrated (P2-2). |
| PowerModels | **Native** | First-class PWL support via SOS2 constraints across all solver backends (P2-2). |
| PowerSimulations | **Native** | Native SOS2 formulation across all thermal formulations and MIP solvers (P2-2). |
| pandapower | **Native** | Native `create_pwl_cost()` API; clean and documented (P2-2). |
| GridCal | Gap | Only polynomial cost model supported; MATPOWER PWL costs are fitted to quadratic on import, losing the piecewise structure (P2-2). |
| MATPOWER* | **Native** | Native PWL cost curves in the MATPOWER case format with direct support in the OPF formulation (P2-2). |

</details>

<details className="eval-details">
<summary>PSS/E RAW File Parsing</summary>

The target full network model (FNM) is distributed in PSS/E RAW format. A tool with
native RAW parsing avoids an error-prone format conversion step and preserves
metadata (switched shunts, multi-section lines, area interchange) that may be lost
in translation. Phase 2 requires support for at least PSS/E v31 format.

| Tool | Rating | Evidence |
|------|--------|---------|
| PyPSA | Gap | No PSS/E import method of any kind. Six available import methods all expect tool-native formats. Phase 2 FNM ingestion requires building a custom converter (P2-1, G-FNM-1). |
| PowerModels | Workaround | Built-in parser supports v33 only; v31 fails on header format mismatch. Parsing v31 files requires header adaptation or parser modification (P2-1, G-FNM-1). |
| PowerSimulations | Workaround | PowerSystems.jl PTI parser supports v33/v35 only; v31 fixed-width column support is absent (P2-1, G-FNM-1). |
| pandapower | Gap | No PSS/E parser of any kind. A production converter is estimated at 2-4 weeks of development effort (P2-1, G-FNM-1). |
| GridCal | Workaround | Parser declares v29-v35 support but is hardcoded to v35 field layout; v31 files fail (P2-1, G-FNM-1). |
| MATPOWER* | **Native** | `psse2mpc()` supports PSS/E Rev 23-33+ natively. The reference DCPF verification confirms correct FNM ingestion at 27,862 buses (P2-1, G-FNM-3). |

</details>

<details className="eval-details">
<summary>Custom Constraint Injection</summary>

Custom constraints are required for modeling market-specific operational limits
(interface flow limits, ramp rate constraints, zonal transfer caps) that are not
part of standard OPF formulations. A tool that exposes the optimization model
directly reduces the effort to add these constraints and extract dual values for
economic analysis.

| Tool | Rating | Evidence |
|------|--------|---------|
| PyPSA | **Native** | `extra_functionality` callback provides full Linopy model access for constraint injection and dual extraction. All 8 extensibility tests pass through documented public APIs (B-1). |
| PowerModels | **Native** | Two-level API (`instantiate_model` + `@constraint` + `optimize_model!`) enables constraint injection in 4 LOC with dual extraction. Clean architectural separation (B-1, B-6). |
| PowerSimulations | **Native** | JuMP model access via `PSI.get_jump_model()` enables arbitrary constraint addition and dual extraction. Requires knowledge of PSI internal constraint naming conventions (B-1). |
| pandapower | Workaround | Requires replicating `_optimal_powerflow`, monkey-patching module references, and capturing the PYPOWER result dict for dual access. Fragile; depends on undocumented internal structure (B-1). |
| GridCal | Workaround | Monkey-patch of `PulpLpModel.solve` relying on internal constraint naming convention `br_flow_upper_lim_0_<idx>`. Fragile; no public API for constraint injection (B-1). |
| MATPOWER* | Native | Documented `mpc.A/l/u` custom constraint API with dual extraction at `results.lin.mu.u.usr`. MIPS fails with user constraints on QP problems, requiring GLPK (LP) with linearized costs (B-1). |

</details>

<details className="eval-details">
<summary>Unit Commitment / Economic Dispatch Pipeline</summary>

A unit commitment and economic dispatch pipeline is the core analytical workflow for
day-ahead market modeling. The tool must support binary commitment variables, minimum
up/down time constraints, and either native two-stage SCUC-then-SCED execution or a
clean mechanism for commitment injection into a dispatch model.

| Tool | Rating | Evidence |
|------|--------|---------|
| PyPSA | **Native** | Native SCUC via `pypsa.optimize` with committable generators, min up/down times, startup costs, and ramp limits as native attributes. SCED achievable via manual `p_min_pu`/`p_max_pu` bound manipulation to fix commitment schedule; stable workaround (A-5, A-6). |
| PowerModels | Gap | No unit commitment capability. UC binary variables, min up/down times, and startup costs are entirely outside the tool's design scope. ED-only SCED is achievable via `replicate()` with ramp constraints (A-5, A-6). |
| PowerSimulations | **Native** | Built-in `ThermalStandardUnitCommitment` formulation with the cleanest separation of concerns among all evaluated tools. UC-to-ED handoff requires manual JuMP variable fixing; no built-in two-stage API (A-5, A-6). |
| pandapower | Gap | No unit commitment formulation. No binary variables, startup costs, or temporal coupling. Single-period economic dispatch only (A-5, A-6). |
| GridCal | **Native** | SCUC via `OpfDispatchMode.UnitCommitment` with ramp rates and min up/down times. SCED achievable via Pmax/Pmin profile manipulation for decommitted generators; stable workaround. Time-series OPF driver is blocked by a TapPhaseControl enum bug in v5.6.28, requiring sequential snapshot execution that loses inter-temporal coupling (A-5, A-6, B-4). |
| MATPOWER* | **Native** | MOST provides built-in SCUC with all constraint types (MinUp/MinDown, startup costs, ramp rates, reserves, storage). Clean two-stage UC/ED via GEN_STATUS toggling and per-period `rundcopf`. A GLPK exit flag integration bug prevents result extraction on non-trivial SCUC problems in the Octave environment (A-5, A-6, A-12). |

</details>

## Phase 2 Readiness Summary

The head-to-head comparison reveals that no single tool covers all six capabilities natively. PyPSA and PowerSimulations each cover four of six natively, but their gaps differ: PyPSA lacks distributed slack OPF and PSS/E parsing, while PowerSimulations lacks SCOPF at scale and PSS/E v31 parsing. PowerModels covers two natively (PWL costs, custom constraints) but lacks UC entirely.

For Phase 2, the critical path capabilities are SCOPF, custom constraint injection, and the UC/ED pipeline. PyPSA is the only tool that covers all three natively (A-9, B-1, A-5, A-6). PowerSimulations covers two of three natively (B-1, A-5/A-6) but requires fragile workarounds for SCOPF at 10,000-bus scale (C-8). PowerModels covers one of three natively (B-1) and requires significant extension code for SCOPF and UC.

PSS/E v31 parsing is a universal gap: no evaluated tool parses v31 files correctly. Phase 2 will require either a custom converter or upstream parser contributions regardless of tool selection.
