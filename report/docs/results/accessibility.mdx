# Accessibility

## Overview

Accessibility measures how easy it is for a new analyst to install, learn, and productively use each tool. The Suite D tests evaluate onboarding friction (D-1: time from install to first successful solve), documentation quality and coverage (D-2), example verification (D-3), error transparency and diagnostic quality (D-4), and code volume required for standard tasks (D-5). This criterion carries **moderate priority** in the rubric; a tool with outstanding expressiveness but poor accessibility creates adoption risk and ongoing maintenance burden for the analyst team.

The six tools separate into three tiers. PyPSA earns Strong with the fastest install-to-first-solve time, working examples, and meaningful error diagnostics. pandapower, PowerModels, and MATPOWER each earn Adequate, with solid core documentation offset by gaps in advanced workflows or platform-specific friction. GridCal and PowerSimulations receive Weak, reflecting documentation gaps, poor OPF diagnostics, and barriers to programmatic use that extend the learning curve well beyond days.

## Per-Tool Analysis

<details className="eval-details" open>
<summary>

### PyPSA — <span className="tier-strong">Strong</span>

</summary>

PyPSA offers the lowest onboarding friction of any evaluated tool. Install-to-first-solve completes in 6.4 seconds via `uv sync` followed by a three-line `import pypsa; n = pypsa.Network(); n.lpf()` invocation (D-1). The pandas-like API means users familiar with the Python data science ecosystem can begin constructing networks immediately; buses are DataFrame rows and components are added via `n.add()` calls that mirror pandas conventions.

Five of ten Suite A tests are implementable from official documentation alone, with two more achievable through trial-and-error (D-2). All six built-in examples load correctly, and three out of three tested examples solve without modification (D-3). Error diagnostics are meaningful for all three deliberate error scenarios tested; no silent failures were observed (D-4). The DataFrame-native data model enables immediate use of the pandas ecosystem for downstream analysis (B-5).

The primary accessibility weakness is in intermediate-to-advanced workflows. Shadow price extraction, PTDF bus ordering, and the mixin architecture are undocumented in the user guide (D-2). The transformer `b` field has dual semantics (shunt vs. series susceptance) that are not documented and can cause silent ACPF divergence. Deferred validation accepts invalid bus references at `n.add()` time, catching errors only at `optimize()` (D-4).

| Test | Status | Key Finding |
|------|--------|-------------|
| D-1 | pass | 0.4s install, 6.4s first solve (post-bytecode cache) |
| D-2 | informational | 50% docs-only implementability for Suite A |
| D-3 | pass | 6/6 examples load, 3/3 solve |
| D-4 | pass | 3/3 error scenarios produce meaningful diagnostics |
| D-5 | informational | Mean 275 LOC across TINY test scripts |

</details>

### pandapower — <span className="tier-adequate">Adequate</span>

</summary>

pandapower's element-based API is intuitive for users with a power systems background. Creating a network reads like assembling a one-line diagram: `pp.create_bus()`, `pp.create_line()`, `pp.create_ext_grid()`. Install-to-first-solve completes in 5.35 seconds with no system dependencies or compiler toolchain (D-1). Core power flow documentation is strong, with three of ten Suite A tests implementable from documentation alone (D-2). Fourteen of sixteen examples run without modification (D-3).

The Adequate tier reflects meaningful gaps beyond the core power flow use case. Documentation never explicitly states what pandapower *cannot* do (no SCUC, no SCOPF, no multi-period OPF), forcing users to infer limits from feature absence (D-2). OPF error diagnostics are poor: `runopp()` returns only a boolean `converged` flag with no infeasibility or convergence progress information (D-4). The `**kwargs` passthrough silently absorbs invalid parameters; `distributed_slack=True` passed to `rundcopp()` is silently ignored (D-4). Two example failures were observed: the getting-started example fails with a zero-length-line divide-by-zero, and the case9 DC OPF silently returns incorrect results (D-3).

|------|--------|-------------|
| D-1 | informational | 5.35s cold install-to-first-solve |
| D-2 | informational | 3/10 from docs only; 7/10 need source |
| D-3 | informational | 14/16 examples pass; 2 failures |
| D-4 | informational | OPF error quality: poor; element creation: good |
| D-5 | informational | 2,185 LOC total Suite A; core API concise |

### PowerModels — <span className="tier-adequate">Adequate</span>

</summary>

PowerModels has well-structured API documentation and a clear mathematical formulation reference that maps JuMP model variables to their power system equivalents. For users already comfortable with Julia and JuMP, the learning curve is manageable. Install-to-first-solve completes in 3.6 seconds with a warm precompile cache (D-1). Eight of ten quickstart examples work on v0.21.5 (D-3). Infeasible OPF problems produce clear `INFEASIBLE` / `INFEASIBLE_POINT` status codes (D-4).

The primary accessibility barrier is the Julia ecosystem itself: first-time users must learn the package manager, wait through 5-15 minutes of JIT compilation on first use, and adapt to Julia's dispatch-based programming model (D-1). Only one of twelve Suite A tests is fully documented; six are partially documented, and two require source code reading (D-2). Error quality is mixed: solver-detectable problems receive clean diagnostics, but invalid `bus_type` values produce silent failures (`OTHER_ERROR`, NaN objective), and missing generator cost data produces a cryptic `KeyError("model")` without generator identification (D-4).

| Test | Status | Key Finding |
|------|--------|-------------|
| D-1 | qualified_pass | 3.6s warm; 5-15 min cold JIT |
| D-2 | qualified_pass | 1/12 fully documented; no source patching needed |
| D-3 | qualified_pass | 8/10 examples pass; PSS/E edge case on non-standard file |
| D-4 | qualified_pass | 1 clear, 1 cryptic, 1 silent failure |
| D-5 | informational | Mean 397 NBNCL; range 176-663 |

<summary>

### GridCal — <span className="tier-weak">Weak</span>

</summary>

GridCal has a clean core API for power flow (`open_file()` / `power_flow()` / `linear_opf()`) and installs via `uv sync` with no compilation, achieving first solve in 1.485 seconds (D-1). However, only two of ten Suite A tests are completable from official documentation alone (D-2). Advanced OPF workflows (SCUC, SCED, SCOPF, multi-period storage) remain underdocumented.

The most significant accessibility concern is OPF diagnostic quality. Infeasible OPF problems report `converged=True`, and silently ignored options produce no warnings (D-4). This asymmetry between power flow diagnostics (which provide excellent convergence feedback including residuals and iteration counts) and OPF diagnostics (which provide none) creates a dangerous trap for users who assume consistent behavior. Rebrand confusion compounds the problem: the PyPI package is `veragridengine`, the import is `VeraGridEngine`, and legacy documentation references `GridCal` (D-1). API naming is inconsistent between solver types: `PowerFlowOptions.solver_type=` vs. `OptimalPowerFlowOptions.solver=` (D-2). The official GridCalTutorials repository is completely broken with deprecated imports; 49 functional examples exist in the main repo but are less discoverable (D-3).

|------|--------|-------------|
| D-1 | informational | 1.485s first solve; rebrand naming confusion |
| D-2 | informational | 2/10 from docs only; OPF options underdocumented |
| D-3 | informational | 49 examples in main repo; tutorials repo broken |
| D-4 | informational | OPF: infeasible reports converged=True; PF: good diagnostics |
| D-5 | informational | Median 203 NBNC lines; temporal profile boilerplate adds 30-50 lines |

<details className="eval-details">
<summary>

### PowerSimulations — <span className="tier-weak">Weak</span>

</summary>

PowerSimulations shares the Julia ecosystem onboarding barrier with PowerModels but adds additional complexity through its layered architecture: users must understand PowerSystems.jl (data model), PowerSimulations.jl (simulation engine), PowerFlows.jl (power flow), and PowerNetworkMatrices.jl (linear analysis) as four interacting packages. Time-to-first-solve is 19 seconds from a warm Julia depot due to JIT compilation across multiple packages (D-1).

Only three of ten Suite A tests are achievable from official documentation alone; critical gaps exist in dual unit conversion (which can produce 100x wrong LMPs), time series multiplier semantics, and the `initialize_model=false` workaround that is necessary for HiGHS but undocumented (D-2). Zero of ten official examples run without modification; all require the external `PowerSystemCaseBuilder` package (D-3). The median code volume is 332 LOC per Suite A test, approximately 6x more than equivalent PyPSA or pandapower tests (D-5).

Infeasibility reporting is best-in-class: automatic model serialization to JSON enables debugging (D-4). Julia's type system catches structurally invalid states early. However, zero-cost generators silently produce `OPTIMAL` status with meaningless dispatch results (D-4), creating a subtle correctness trap.

|------|--------|-------------|

| D-2 | informational | 3/10 from docs; dual unit conversion undocumented |
| D-3 | informational | 0/10 examples run unmodified; require PowerSystemCaseBuilder |
| D-4 | informational | Infeasibility: best-in-class; zero-cost: silent failure |
| D-5 | informational | Median 332 LOC; ~6x PyPSA/pandapower equivalent |

</details>

<summary>

### MATPOWER — <span className="tier-adequate">Adequate</span> (Reference Only)

</summary>

:::note[Reference Implementation]
MATPOWER is excluded from the primary ranking because the customer requires inspectable source code, which precludes the compiled MATLAB runtime. Grades reflect evaluation performance only.
:::

MATPOWER benefits from over two decades of academic use, with a comprehensive 247-page User's Manual and a large body of example scripts. The `runpf()` and `rundcopf()` API is among the most compact in the evaluation, requiring only three lines for a first solve (D-1). Five of ten Suite A tests are completable from documentation alone (D-2). Six of seven tested examples run without modification in Octave (D-3). Data validation errors for bus types and missing fields are clear and actionable (D-4).

The primary accessibility limitations are the interactive installer (`install_matpower.m`) that blocks on stdin and is unusable in CI or batch environments (D-1), hidden path dependencies not documented outside the interactive installer, and the MOST scheduling extension whose table formats (xgd_table, sd_table, profile structs) require reading source code (D-2). The Octave platform lacks DataFrame abstractions, adding 10-20 lines per CSV export compared to Python or Julia (D-5). GLPK solver error messages give no indication when a feasible solution exists (D-4).

## Summary

Accessibility separates the tools into three clear bands. PyPSA's combination of sub-7-second onboarding, working examples, and informative error messages makes it the most accessible tool for a new analyst (D-1, D-3, D-4). pandapower, PowerModels, and MATPOWER each provide solid foundations for core workflows but degrade for advanced use cases: pandapower's OPF diagnostics are poor (D-4), PowerModels imposes the Julia JIT compilation barrier (D-1), and MATPOWER's MOST extension requires source code reading (D-2). GridCal and PowerSimulations present the steepest learning curves: GridCal's OPF silently reports success on infeasible problems (D-4), and PowerSimulations requires knowledge of four interacting packages with zero working examples (D-3).
