## Scope Rationale

### Proprietary Tool Scope

The SOW specifies evaluation of "open-source and proprietary" technologies. This evaluation
focused exclusively on open-source tools. This was a deliberate scope decision driven by the
program's deployment requirements.

The **Supply Chain** criterion serves as a gate: a grade of Weak or Failing is disqualifying
regardless of performance on all other criteria (see
[Use Cases & Criteria](/use-cases-criteria#supply-chain-gate-criterion)). This gate exists because
the customer requires inspectable source code for every component in the execution stack.

Any proprietary tool, by definition, contains non-inspectable components. A proprietary tool
would fail the Supply Chain gate and be disqualified before other criteria are evaluated.
MATPOWER's exclusion from primary ranking demonstrates this principle: despite strong technical
performance, its dependency on MATLAB's compiled runtime makes it unsuitable for recommendation
because the customer requires source code they can inspect. MATPOWER is retained as a reference
benchmark precisely because it illustrates the boundary: technically capable but architecturally
excluded. MATPOWER's technical results are available in the [Supply Chain](/results/supply-chain)
and [Tools Evaluated](/tools-evaluated) pages for reference.

Evaluating proprietary tools that would be immediately disqualified by the gate criterion would
consume evaluation resources without producing actionable recommendations. The six tools
selected for evaluation represent the complete set of mature, open-source power system modeling
tools that could plausibly pass the Supply Chain gate.

### Forecasting Scope

The SOW references "support for long-term forecasting." The tools under evaluation are power
system *analysis* engines; they solve power flow, optimal power flow, unit commitment, and
contingency analysis. They consume forecasts as inputs but do not produce them.

"Support for forecasting" in this context means the tool's ability to accept scenario-based
inputs over multi-period time horizons and produce meaningful results across those scenarios.
This capability is directly tested by:

- **A-5 (SCUC):** 24-hour security-constrained unit commitment with hourly time steps, requiring
  the tool to accept and optimize over a time-series load forecast.
- **B-4 (Stochastic scenario wrapping):** The tool's ability to wrap deterministic solves in
  scenario loops with correlated input variations, the mechanism by which forecast uncertainty
  is propagated through the analysis.

The forecasting engine itself (load forecasting, renewable generation forecasting, price
forecasting) is a separate capability that will be integrated in Task 2 model development,
using the selected tool's scenario ingestion interface.

### Vulnerability Identification Scope

The SOW references "identifying vulnerabilities and pressure points" in the grid. This maps
directly to **contingency analysis**: the systematic identification of conditions under which
equipment failure leads to constraint violations, overloads, or voltage collapse.

The evaluation tests this capability through:

- **A-7 (N-M Contingency Sweep):** Enumerate and evaluate N-1 and N-M contingencies to
  identify which equipment failures cause violations, that is, the grid's vulnerability points.
- **A-9 (Security-Constrained OPF):** Optimize generation dispatch subject to contingency
  constraints, ensuring the system remains feasible under the identified vulnerabilities.
- **B-3 (Contingency loop construction):** The extensibility to build custom contingency
  screening pipelines that prioritize vulnerabilities by severity and likelihood.

The term "pressure points" maps to binding constraints identified through SCOPF and contingency
screening: locations where the network is most stressed under credible failure scenarios. The
[Head-to-Head comparison](/results/head-to-head) and
[Expressiveness results](/results/expressiveness) document each tool's capability to perform
this analysis.
