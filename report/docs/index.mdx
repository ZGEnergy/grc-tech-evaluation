# Phase 1 Tool Selection Report

Six open-source power system modeling tools — PyPSA, PowerModels, PowerSimulations,
pandapower, GridCal, and MATPOWER — were evaluated against a six-criterion rubric
(Expressiveness, Extensibility, Scalability, Accessibility, Maturity, Supply Chain) using
standardized test suites on ACTIVSg 2k-bus and 10k-bus reference networks with open-source
solvers only. Supply Chain serves as a gate criterion; the remaining five criteria are ranked
by lexicographic priority with Expressiveness as the primary differentiator. MATPOWER is
designated reference benchmark only and excluded from primary ranking due to its proprietary
MATLAB runtime dependency.

**We recommend PyPSA** as the selected tool for Phase 2 development. PyPSA wins decisively
on the primary criterion: its Expressiveness grade of B+ is a full grade step above the B-
shared by the next two eligible tools. Beyond this differentiator, PyPSA has the strongest
balanced profile among all candidates — A- in both Extensibility and Accessibility,
A in Maturity, and B+ in Supply Chain. It demonstrated first-class preventive SCOPF via
`optimize_security_constrained()`, custom constraint injection in 2 lines of code via
Linopy model split, and a mature multi-contributor development community backed by
institutional funding at TU Berlin and KIT. The Python ecosystem ensures analysts can
integrate PyPSA into existing workflows without adopting a niche language runtime.

The recommendation is robust: PyPSA holds the #1 position in three of four alternative
ranking scenarios tested (extensibility-first, maturity-gate, scalability-first); Scenario 3
(Scalability as top priority) is the only exception, where PyPSA's C+ Scalability drops it to
#4 behind GridCal — a tool whose C Maturity and blocking battery sign error make it unsuitable
for recommendation regardless. Five risks require active mitigation in Phase 2, with PSS/E
ingestion and PWL cost curve gaps rated HIGH severity.

## Why PyPSA

1. **Expressiveness leadership.** PyPSA's B+ Expressiveness is a full grade step above the
   B- tier shared by PowerModels and PowerSimulations. It demonstrated native preventive
   SCOPF (A-9), native PTDF/LODF matrix computation (B-9), and the broadest coverage of
   standard power system analyses among all evaluated tools — without requiring user-assembled
   solver code for each analysis type.

2. **Balanced profile across non-primary criteria.** PyPSA earned A- in both Extensibility and
   Accessibility, A in Maturity, and C+ in Scalability. The A- Extensibility reflects a clean
   Linopy model-split API for custom constraints, and A Maturity reflects a healthy
   multi-contributor development community with institutional backing. Scalability (C+) is the
   known weak spot — driven by the HiGHS single-threaded MILP timeout on SCUC — and is
   addressed in the risk register (R3).

3. **Python ecosystem and analyst accessibility.** PyPSA is pure Python, built on the
   pandas/numpy/scipy stack that many operational analysts already know. This eliminates the
   language-adoption barrier that Julia (PowerModels, PowerSimulations) or MATLAB (MATPOWER)
   would impose. Integration with existing data pipelines, visualization tools, and notebook
   workflows is immediate rather than requiring bridge infrastructure.

## Runner-Up: PowerModels

PowerModels ranks second on the strength of its A- Extensibility and its JuMP foundation,
which provides the most flexible constraint injection API among all evaluated tools. It
demonstrated native PSS/E v33 parsing, native piecewise-linear cost curves, and PTDF
computation with sub-1e-11 error at 10k-bus. However, PowerModels was not selected for three
reasons: its B- Expressiveness reflects the cost of JuMP flexibility — 6 of 11 Suite A tests
required user-assembled code averaging 269 lines per test; its B Maturity with a bus factor
of 1 (82.4% of commits from a single contributor) signals sustainability risk; and Julia
remains a niche language for many analyst teams. PowerModels should be reconsidered if
PyPSA's SCUC scalability limitation proves unresolvable.

| Risk | Severity | Mitigation |
|------|----------|------------|
| **R1 — No PSS/E RAW ingestion:** PyPSA has no PSS/E RAW ingestion path; the FNM is delivered in v31 format (G-FNM-1 fail). Phase 2 cannot start without a conversion pipeline. | HIGH | Develop a PSS/E v31 → MATPOWER .m converter in Phase 2 pre-work using the shared `matpower_loader` that already passes G-FNM-3. Estimated 1–2 weeks. Validate against the MATPOWER DCPF baseline before any PyPSA optimization run. |
| **R2 — No native PWL cost curves:** PyPSA has no native PWL cost curve support (tracked as issue #1020). Accurate LMP decomposition requires convex piecewise-linear heat rates. | HIGH | Implement PWL costs via `extra_functionality` using SOS2 binary variables injected into the HiGHS MILP. Prototype and validate against PowerSimulations' native SOS2 output before committing to the architecture. Estimated 1–2 weeks. |
| **R3 — SCUC SMALL timeout:** SCUC at SMALL scale fails due to HiGHS single-threaded MILP timeout (C-4). Production UC runs at the target scale will be larger. | MED | Evaluate Gurobi or CPLEX as a drop-in HiGHS replacement for MILP workloads — PyPSA's solver interface is a single parameter change. If a commercial solver is unavailable in the classified environment, implement a Lagrangian relaxation pre-solve to warm-start HiGHS. |
| **R4 — Shadow price assignment bug:** `lines_t.mu_upper` and `lines_t.mu_lower` are silently empty after `n.optimize()`. Congestion pricing logic in Phase 2 depends on binding constraint duals. | MED | Use `n.statistics()` to extract shadow prices post-solve and validate against manually computed dual variables on a small reference network before integrating into LMP decomposition. File an upstream bug report. |
| **R5 — Transformer b field dual semantics:** PyPSA's transformer `b` field carries dual semantics (susceptance vs. tap-ratio) that are undocumented. Incorrect FNM parameterization will produce silently wrong power flows. | LOW | Establish a transformer round-trip test: load FNM, run DCPF, compare branch flows to the MATPOWER baseline used in G-FNM-3. Enforce this as a regression gate before any production optimization run. |
