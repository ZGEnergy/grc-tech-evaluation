# Phase 1 Tool Selection Report

Six open-source power system modeling tools (PyPSA, PowerModels.jl, PowerSimulations.jl, pandapower, GridCal, and MATPOWER) were evaluated against a six-criterion rubric: Expressiveness, Extensibility, Scalability, Accessibility, Maturity, and Supply Chain. Testing used standardized test suites on 39-bus, 2,000-bus, and 10,000-bus ACTIVSg reference networks with open-source solvers only. MATPOWER served as a reference benchmark but is excluded from the primary ranking because the customer requires inspectable source code, which precludes MATLAB's compiled runtime.

**PyPSA is the recommended tool for Phase 2 development.** It is the only tool to earn Strong in Expressiveness, the primary differentiating criterion, and it holds the top position across all three sensitivity scenarios tested.

## Why PyPSA

1. **Expressiveness leadership.** PyPSA is the only tool graded Strong in Expressiveness. It demonstrated native preventive SCOPF via `optimize_security_constrained()` (A-9), native PTDF/LODF matrix computation (B-9), and the broadest coverage of standard power system analyses among all evaluated tools. No other candidate matched this breadth without requiring user-assembled solver code for each analysis type.

2. **Strong, balanced profile.** Beyond the primary criterion, PyPSA earned Strong in Extensibility, Accessibility, Maturity, and Supply Chain, with Adequate in Scalability. The Strong Extensibility reflects a clean extra_functionality callback for custom constraint injection (B-1). The Strong Maturity reflects a healthy multi-contributor development community with institutional backing. Scalability (Adequate) is the one area requiring active Phase 2 attention; Linopy model-building overhead at 10,000 buses (302 seconds vs. 6 seconds of solver time, C-3) is addressed in the risk register (R1).

3. **Python ecosystem accessibility.** PyPSA is pure Python, built on the pandas/numpy/scipy stack. This eliminates the language-adoption barrier that Julia-based tools (PowerModels.jl, PowerSimulations.jl) would impose. Integration with existing data pipelines, visualization tools, and notebook workflows requires no bridge infrastructure.

## Runner-Up: PowerModels.jl

PowerModels.jl ranks second on the strength of its Strong Extensibility and its JuMP foundation, which provides the most flexible constraint injection API among all evaluated tools (B-1). It demonstrated native piecewise-linear cost curves and PTDF computation with sub-1e-11 error at 10,000 buses (B-9). PowerModels.jl was not selected for three reasons: its Adequate Expressiveness reflects the cost of JuMP flexibility, with 6 of 11 Suite A tests requiring user-assembled code averaging 269 lines per test; its Adequate Maturity carries a bus factor of 1 (82.4% of commits from a single contributor), signaling sustainability risk; and Julia remains a niche language for many analyst teams. PowerModels.jl should be reconsidered if PyPSA's model-building overhead at production scale (R1) proves unresolvable.

<summary>Risk Register (5 items)</summary>

| Risk | Severity | Mitigation |
|------|----------|------------|
| **R1: Linopy model-building overhead at production scale.** PyPSA's Linopy layer takes 302 seconds to construct the optimization model for a 10,000-bus DC OPF (C-3), compared to 6 seconds of solver time. At production scale with repeated solves, this overhead may dominate wall-clock time. | HIGH | Profile Linopy model construction during Phase 2 development. If overhead is prohibitive, evaluate direct solver interface (HiGHS C API) for the inner loop, using PyPSA only for network data management and result post-processing. The solver-only time (6.2s on 10,000 buses) is competitive. |
| **R2: No native PSS/E RAW file parser.** PyPSA has no PSS/E import capability (P2-1). Phase 2 requires ingesting the full network model distributed in PSS/E RAW format. A custom converter must be built or an existing parser must be adapted. | HIGH | Build a PSS/E RAW-to-PyPSA converter during Phase 2. The FNM intermediate format (CSV-based) already exists and was used for Phase 1 FNM testing. Alternatively, use an external parser as a preprocessing step and ingest via the existing MATPOWER .m importer. |
| **R3: Distributed slack OPF not available in optimization context.** PyPSA supports distributed slack in power flow but not in OPF formulations (A-11, partial pass). LMP decomposition without distributed slack produces systematically different values from published data. | MED | Implement distributed slack as a custom Linopy constraint via PyPSA's `extra_functionality` callback. The PTDF formulation with load-proportional weights is well-documented. Estimated effort: weeks. |
| **R4: Single-threaded HiGHS solver limits parallelism.** HiGHS simplex does not parallelize; SCOPF on a 10,000-bus network showed 0.92x speedup with 32 threads (C-8). Production contingency analysis will not benefit from multi-core hardware through solver-internal threading. | MED | Use process-level parallelism for independent contingency sub-problems rather than relying on solver-internal threading. PyPSA's architecture supports independent model instantiation per contingency group. |
| **R5: Piecewise-linear cost curves require tranche workaround.** PyPSA does not natively support piecewise-linear cost curves (P2-2, partial). The workaround (generator tranche decomposition) approximates PWL behavior but introduces modeling complexity. | MED | Implement the generator tranche decomposition as a reusable utility during Phase 2. Validate against reference PWL solutions from a tool with native SOS2 support to ensure numerical fidelity. |
