# Phase 1 Tool Selection Report

Six open-source power system modeling tools (PyPSA, PowerModels.jl, PowerSimulations.jl, pandapower, GridCal, and MATPOWER) were evaluated against a six-criterion rubric: Expressiveness, Extensibility, Scalability, Accessibility, Maturity, and Supply Chain. Testing used standardized test suites on 39-bus, 2,000-bus, and 10,000-bus ACTIVSg reference networks with open-source solvers only. MATPOWER served as a reference benchmark but is excluded from the primary ranking because the customer requires inspectable source code, which precludes MATLAB's compiled runtime.

**PyPSA is the recommended tool for Phase 2 development.** It is the only tool to earn Strong in Expressiveness, the primary differentiating criterion, and it holds the top position across all three sensitivity scenarios tested.

## Why PyPSA

1. **Expressiveness leadership.** PyPSA is the only tool graded Strong in Expressiveness. It demonstrated native preventive SCOPF via `optimize_security_constrained()`, native PTDF/LODF matrix computation, and the broadest coverage of standard power system analyses among all evaluated tools. No other candidate matched this breadth without requiring user-assembled solver code for each analysis type.

2. **Strong, balanced profile.** Beyond the primary criterion, PyPSA earned Strong in Extensibility, Accessibility, Maturity, and Supply Chain, with Adequate in Scalability. The Strong Extensibility reflects a clean extra_functionality callback for custom constraint injection. The Strong Maturity reflects a healthy multi-contributor development community with institutional backing. Scalability (Adequate) is the one area requiring active Phase 2 attention; Linopy model-building overhead at 10,000 buses (302 seconds vs. 6 seconds of solver time) is addressed in the risk register (R1).

3. **Python ecosystem accessibility.** PyPSA is pure Python, built on the pandas/numpy/scipy stack. This eliminates the language-adoption barrier that Julia-based tools (PowerModels.jl, PowerSimulations.jl) would impose. Integration with existing data pipelines, visualization tools, and notebook workflows requires no bridge infrastructure.

## Runner-Up: PowerModels.jl

PowerModels.jl ranks second on the strength of its Strong Extensibility and its JuMP foundation, which provides the most flexible constraint injection API among all evaluated tools. It demonstrated native piecewise-linear cost curves and PTDF computation with sub-1e-11 error at 10,000 buses. PowerModels.jl was not selected for three reasons:

1. **Adequate Expressiveness reflects the cost of JuMP flexibility.** Half of the expressiveness tests required user-assembled JuMP code averaging 487 lines per test, compared to PyPSA's native API calls.
2. **Adequate Maturity carries sustainability risk.** 82.4% of commits come from a single contributor (bus factor of 1), raising concerns about long-term maintenance.
3. **Julia adoption barrier.** Julia remains a niche language for many analyst teams, imposing a workforce ramp-up cost that Python-based tools avoid.

PowerModels.jl should be reconsidered if PyPSA's model-building overhead at production scale (R1) proves unresolvable.

<summary>Phase 2 Development Scope</summary>

### Stage 1 — Historical Calibration

Assemble the full network model and supporting documents to establish topology. Data gathering includes available generators, third-party vendor data, flow data from third-party sensors, and constraint setup and binding information from the ISO. Intertie actual flows may be available depending on whether the customer can navigate a regulatory barrier (the ISO restricts this data to Transmission Operators).

The goal is to calibrate a state estimate — effectively rebuilding state estimation since the ISO does not publish it — and to calibrate distribution factors and learn stochastic scenarios for generation outages and renewable variability.

**Success criterion:** constraints firing as published per ISO market results.

### Stage 2 — Real-Time Estimation

Inputs: real-time sensor data, aggregated real-time load and generation, and published shadow prices for constraints. Shadow prices combined with constraint setup information indicate whether a line is binding, providing at minimum a floor on flow levels. From these inputs, create a real-time state estimation for the entire grid.

### Stage 3 — Forecasting

Use the learned parameters from Stages 1–2 (distribution factors, stochastic scenarios around generation outages and renewables). At this stage no prices are available — inputs are limited to load forecast, renewable forecast, and known generation and topology for the next few days. Produce a range of possible power flow and congestion outcomes over a 2–3 day horizon.

### Supporting Work

- ETL pipelines for all data sources across all three stages
- Distribution factor calibration down to bus level (more granular than the substation-level commitment in the SOW)
- **Deliverable:** a forward-looking vulnerability assessment tool
