# Tools Evaluated

This page profiles the six open-source power-system modeling tools evaluated under the Phase 1 rubric. Each tool was tested against six criteria: Expressiveness, Extensibility, Scalability, Accessibility, Maturity, and Supply Chain. Supply Chain serves as a gate criterion; any tool graded below Adequate is disqualified.

Tools are ordered by final rank. MATPOWER is included as a reference benchmark only and is excluded from the primary ranking. See the [exclusion rationale](#matpower-exclusion) at the bottom of this page for details.

### #1 - PyPSA

</summary>

> Python for Power System Analysis. A DataFrame-native power system modeling framework
> with native support for DCPF, ACPF, DC OPF, SCUC, SCOPF, lossy OPF, and multi-period
> storage optimization.

| | |
|---|---|
| **Language / Runtime** | Python 3.9+ |
| **Solver Ecosystem** | HiGHS, GLPK, SCIP, Gurobi, CPLEX (via Linopy) |
| **Maintainer** | TU Berlin / Open Energy Transition |
| **License** | MIT |

#### Grade Profile

| Criterion | Grade |
|-----------|-------|
| Expressiveness | Strong |
| Extensibility | Strong |
| Scalability | Adequate |
| Accessibility | Strong |
| Maturity | Strong |
| Supply Chain | Strong |

#### Key Strengths

1. **Broadest native expressiveness.** PyPSA passes 8 of 11 expressiveness tests across all target problem types, including lossy DC OPF with LMP decomposition (A-10) and multi-period storage OPF (A-12). It is the only tool to achieve Strong on the highest-priority criterion.
2. **Zero-friction extensibility.** The Linopy `extra_functionality` callback enables custom constraint injection with full dual extraction in minimal code. All 8 extensibility tests pass (B-1 through B-9), including solver swap, PTDF extraction, and graph-based network analysis.

#### Key Weaknesses

1. **No PSS/E format ingestion.** PyPSA has no native capability to parse PSS/E RAW files. Phase 2 FNM ingestion will require building a custom converter (P2-1).
2. **Linopy model-building overhead at scale.** On the 10,000-bus network, DC OPF model construction takes 302 seconds compared to 6 seconds of solver time (C-3). This overhead, not solver performance, is the scalability bottleneck.

#### Notable Workarounds

- **A-6 (SCUC commitment fix):** Manual `p_min_pu`/`p_max_pu` bound construction to correct commitment schedule. Durability: stable.
- **A-11 (Distributed slack OPF):** No formulation exists in the OPF context. Durability: blocking.
- **F-3 (GPL dependency):** The Levenshtein package (GPL, used only for developer UX) is replaceable with the MIT-licensed rapidfuzz. Durability: stable.

#### Phase 2 Viability

PyPSA is the recommended tool for Phase 2. Its primary risk is the PSS/E ingestion gap (P2-1), which requires a custom converter to load the full network model. The Linopy overhead at 10,000-bus scale (C-3) may require profiling and optimization for production workflows but does not block feasibility. All other Phase 2-critical capabilities are natively supported or achievable through stable workarounds.

</details>

### #2 - PowerModels.jl

</summary>

> A Julia/JuMP-based power network optimization framework from Los Alamos National
> Laboratory with a clean four-layer dispatch architecture enabling custom constraint
> injection in minimal code.

| | |
|---|---|
| **Language / Runtime** | Julia 1.6+ |
| **Solver Ecosystem** | HiGHS, Ipopt, GLPK, SCIP, Gurobi, CPLEX (via JuMP/MOI) |
| **Maintainer** | Los Alamos National Laboratory (LANL) |
| **License** | BSD-3-Clause |

#### Grade Profile

| Criterion | Grade |
|-----------|-------|
| Expressiveness | Adequate |
| Extensibility | Strong |
| Scalability | Adequate |
| Accessibility | Adequate |
| Maturity | Adequate |
| Supply Chain | Adequate |

#### Key Strengths

1. **Clean extensibility architecture.** The two-level API (`instantiate_model`/`optimize_model!`) exposes the JuMP model for arbitrary constraint injection in 4 lines of code with full dual extraction (B-1, B-6).
2. **Machine-epsilon PTDF accuracy.** PTDF flow prediction error is below 1e-11 at 10,000-bus scale, with DC OPF completing in 4 seconds across 4 solvers (B-9, C-3).

#### Key Weaknesses

1. **No native unit commitment.** SCUC is entirely outside the tool's design scope (A-5). Any unit commitment formulation must be built from scratch in JuMP.
2. **ACPF non-convergence at scale.** ACPF fails at 10,000-bus scale due to Ipopt/MUMPS divergence (C-2). The primary author has been inactive for 12 months, creating bus factor risk (E-6).

#### Notable Workarounds

- **A-9 (SCOPF):** Manual Benders cutting plane with LODF constraint injection. Durability: stable.
- **A-11 (Distributed slack):** Requires approximately 150 lines of custom PTDF-based JuMP OPF code. Durability: blocking.
- **C-3 (10,000-bus DC OPF):** Quadratic cost linearization required for 45.5% of generators. Durability: stable.

#### Phase 2 Viability

PowerModels.jl is a viable Phase 2 candidate with caveats. The absence of native SCUC (A-5) means all unit commitment logic must be hand-built in JuMP, adding significant development effort. ACPF non-convergence at 10,000-bus scale (C-2) and the primary maintainer's inactivity (E-6) are risks that would need mitigation through solver tuning and dependency on the secondary maintainer (odow).

</details>

<summary>

### #3 - PowerSimulations.jl

</summary>

> An NREL/DOE simulation-oriented optimization framework built on JuMP with built-in
> unit commitment formulations and a 5-layer package architecture.

| | |
|---|---|
| **Language / Runtime** | Julia 1.6+ |
| **Solver Ecosystem** | HiGHS, SCIP, Gurobi, CPLEX (via JuMP/MOI) |
| **Maintainer** | National Renewable Energy Laboratory (NREL) |
| **License** | BSD-3-Clause |

#### Grade Profile

| Criterion | Grade |
|-----------|-------|
| Expressiveness | Adequate |
| Extensibility | Strong |
| Scalability | Adequate |
| Accessibility | Weak |
| Maturity | Adequate |
| Supply Chain | Adequate |

#### Key Strengths

1. **Built-in unit commitment formulations.** ThermalStandardUnitCommitment provides the cleanest separation of device formulations among all evaluated tools, with native ramp limits and min up/down time constraints (A-5, B-6).
2. **Strong institutional backing.** NREL/DOE funding through federal appropriations provides the most durable funding model among evaluated tools (E-4, E-6).

#### Key Weaknesses

1. **Fragile scalability at 10,000-bus.** DC OPF at 10,000-bus scale requires 5 stacked workarounds (initialization bypass, linearized costs, all generators available, unbounded branches, no hydro). SCOPF crashes with HiGHS OTHER_ERROR on the 10,000-bus network (C-3, C-8).
2. **Severe documentation gaps.** Zero of 10 official examples run unmodified. Only 3 of 10 test scenarios are achievable from documentation alone, with a median of 332 lines of code per test (D-2, D-3, D-5).

#### Notable Workarounds

- **A-5 (SCUC initialization):** `initialize_model=false` with direct `JuMP.optimize!()` to bypass broken initialization. Durability: fragile.
- **A-12 (Storage dispatch):** Manual JuMP BESS variables and constraints depending on PSI internal constraint naming conventions. Durability: fragile.
- **C-8 (SCOPF at scale):** Top-50 contingency selection with LODF constraint injection. Durability: fragile.

#### Phase 2 Viability

PowerSimulations.jl has the strongest native unit commitment capability but faces significant Phase 2 risks. The 5-workaround stack for 10,000-bus OPF (C-3) and SCOPF crash at scale (C-8) indicate architectural fragility. Documentation deficiencies (D-2, D-3, D-5) would impose high onboarding costs. The Weak Accessibility grade reflects a steep learning curve that would slow Phase 2 development velocity.

</details>

### #4 - GridCal

</summary>

> A Python power system simulation tool with native LODF-based SCOPF and parameter-only
> solver swap via PuLP.

| | |
|---|---|
| **Language / Runtime** | Python 3.10+ |
| **Solver Ecosystem** | HiGHS, SCIP (via PuLP); internal Newton-Raphson for PF |
| **Maintainer** | Santiago Penate Vera (individual) |
| **License** | MPL-2.0 |

#### Grade Profile

| Criterion | Grade |
|-----------|-------|
| Expressiveness | Adequate |
| Extensibility | Adequate |
| Scalability | Adequate |
| Accessibility | Weak |
| Maturity | Weak |
| Supply Chain | Strong |

#### Key Strengths

1. **Native LODF-based SCOPF.** The `consider_contingencies` parameter activates SCOPF with parameter-only solver swap via PuLP (A-9, C-7).
2. **Fast power flow at scale.** DCPF and ACPF scale to the 10,000-bus network with first-class PTDF/LODF extraction (C-1, C-2, B-9).

#### Key Weaknesses

1. **Bus factor of 1 with non-functional CI.** 70.3% of lifetime commits come from a single maintainer with no institutional backing. The CI pipeline runs only CodeQL; the test suite is never executed in automation (E-2, E-3, E-6).
2. **Blocking formulation bugs.** Battery energy balance has a sign error (A-12) and distributed slack is hardcoded off in OPF mode (A-11). Neither has a workaround.

#### Notable Workarounds

- **A-11 (Distributed slack OPF):** Hardcoded off in OPF; no workaround available. Durability: blocking.
- **A-12 (Battery storage):** Energy balance sign error; no workaround available. Durability: blocking.
- **B-1 (Custom constraints):** Monkey-patching PulpLpModel.solve with internal naming conventions. Durability: fragile.
- **B-4 (Multi-period OPF):** Sequential snapshot OPF to avoid TapPhaseControl enum bug. Durability: stable.

#### Phase 2 Viability

GridCal presents significant Phase 2 risks. Two blocking bugs (A-11, A-12) have no workaround and would require upstream source code fixes. The bus factor of 1 (E-6) and absence of functional CI (E-3) create sustainability concerns. The Weak Maturity grade reflects a project that depends entirely on a single individual for continued development.

</details>

<summary>

### #5 - pandapower

</summary>

> A mature Python power flow tool with a DataFrame-centric data model and embedded
> PYPOWER solver, excelling at steady-state network analysis.

| | |
|---|---|
| **Language / Runtime** | Python 3.10+ |
| **Solver Ecosystem** | PYPOWER interior point (embedded); no external solver swap |
| **Maintainer** | University of Kassel / Fraunhofer IEE |
| **License** | BSD-3-Clause |

#### Grade Profile

| Criterion | Grade |
|-----------|-------|
| Expressiveness | Weak |
| Extensibility | Adequate |
| Scalability | Weak |
| Accessibility | Adequate |
| Maturity | Strong |
| Supply Chain | Strong |

#### Key Strengths

1. **Exemplary supply chain posture.** BSD license, Sigstore provenance, embedded solver, all permissive dependencies, and the smallest dependency footprint (38 packages) among evaluated tools. Fully air-gap installable (F-1 through F-9).
2. **Strongest maturity profile.** Dual institutional backing from University of Kassel and Fraunhofer IEE, with confirmed production deployments at European distribution system operators (E-1 through E-7).

#### Key Weaknesses

1. **Four independent expressiveness failures.** No unit commitment, no multi-period optimization, no lossy DC OPF, and no SCOPF (A-5, A-9, A-10, A-12). The tool cannot express market-operations optimization problems.
2. **Hardcoded solver with no swap mechanism.** The embedded PYPOWER PIPS solver cannot be replaced. It consumes 62 seconds and 2.35 GB on 10,000-bus DC OPF, and encounters out-of-memory failures on SCOPF (C-3, C-7, C-8).

#### Notable Workarounds

- **A-6 (SCUC approximation):** Sequential single-period DCOPF with manual ramp-rate bound manipulation. Durability: fragile.
- **A-9 (SCOPF):** Manual PTDF/LODF with scipy.optimize.linprog, bypassing pandapower OPF entirely. Durability: blocking.
- **B-1 (Custom constraints):** Monkey-patching PYPOWER internals for constraint injection. Durability: fragile.

#### Phase 2 Viability

pandapower is not viable for Phase 2. Four blocking expressiveness gaps (A-5, A-9, A-10, A-12) mean the tool cannot formulate the optimization problems required by the contract. The hardcoded solver (C-7) precludes performance improvement through solver selection. These are architectural limitations, not missing features that could be added through extensions.

</details>

### Reference - MATPOWER

</summary>

:::note[Reference Benchmark Only]
MATPOWER is included as a reference benchmark and is **excluded from primary ranking**.
The customer requires inspectable source code, which precludes the compiled MATLAB
runtime. All MATPOWER testing used GNU Octave.
:::

> The canonical open-source power system analysis toolbox, providing the reference
> MATPOWER case file format used by all other evaluated tools as test input.

| | |
|---|---|
| **Language / Runtime** | MATLAB / GNU Octave |
| **Solver Ecosystem** | MIPS (bundled), GLPK; HiGHS requires MEX compilation |
| **Maintainer** | Ray Zimmerman (independent, formerly Cornell) |
| **License** | BSD-3-Clause |

#### Grade Profile

| Criterion | Grade |
|-----------|-------|
| Expressiveness | Adequate |
| Extensibility | Strong |
| Scalability | Weak |
| Accessibility | Adequate |
| Maturity | Adequate |
| Supply Chain | Strong |

#### Key Strengths

1. **Best-in-class supply chain.** Zero external dependencies, BSD-3-Clause license, 100% pure MATLAB/Octave code, and full air-gap installability. The smallest possible supply chain surface area (F-1 through F-9).
2. **Native PSS/E parsing and built-in SCUC.** The `psse2mpc` converter, native piecewise-linear costs, and MOST extension provide built-in SCUC, multi-period, and storage capabilities (P2-1, P2-2, A-5, A-12).

#### Key Weaknesses

1. **GLPK integration bug blocks SCUC results.** GLPK exit flag mapping incorrectly classifies MIP gap solutions as failures, cascading to all 10,000-bus scalability tests being skipped (A-5, C-4).
2. **Bus factor of 1.** 97.6% of lifetime commits come from a single maintainer, with funding in transition from academic grants (E-6).

#### Notable Workarounds

- **A-9 (SCOPF):** Benders iteration with LODF constraint injection; N-1 feasibility and MIPS numerical issues remain. Durability: blocking.
- **A-10 (Lossy DC OPF):** Iterative loss injection external to OPF; cannot produce non-zero loss LMP components. Durability: blocking.
- **A-5 (SCUC exit code):** GLPK MIP gap exit code mapped to failure; validated on bundled test case. Durability: stable.

#### Exclusion Rationale {#matpower-exclusion}

MATPOWER is excluded from the primary ranking because the customer requires inspectable source code. The MATLAB runtime is a compiled binary that does not meet this requirement. All evaluation testing was conducted using GNU Octave, an open-source MATLAB-compatible interpreter. Octave introduces interpreter-specific limitations: HiGHS MEX bindings are unavailable, performance overhead is measurable on data structure operations, and no production deployment precedent exists for Octave-based power system analysis at scale. MATPOWER's grades are retained in the comparison table to provide a calibration baseline for power systems professionals but do not factor into the tool selection ranking or recommendation.

</details>

## Tools Considered but Ruled Out

The following tools were identified during the initial landscape survey and considered for inclusion in the primary evaluation. Each was excluded for the reason noted.

| Tool | Reason Excluded |
|------|----------------|
| **PYPOWER** | Python port of early MATPOWER; functionally superseded by pandapower, which is faster and better maintained. |
| **GridDyn** | C++ transmission dynamics simulator (LLNL). Last substantive activity circa 2019; project appears stalled. |
| **PowSyBl** | Java-based framework used by European TSOs (RTE, ENTSO-E). Strong CGMES/CIM support but European data model focus; PSS/E RAW ingestion less tested. Misaligned with US transmission work. |
| **PowerGridModel** | C++ core with Python API (Alliander / LF Energy). Designed for high-speed batch power flow in distribution systems; transmission use is possible but not the design target. |
| **HELICS** | DOE co-simulation framework (NREL/PNNL/LLNL). Orchestrates coupling between separate tools — not a solver itself. Not applicable to Phase 1 scope. |
| **GridLAB-D** | Distribution system simulator (PNNL). Scope is explicitly distribution-level; not applicable to high-voltage transmission. |
| **OpenDSS** | Distribution system simulator (EPRI). Same exclusion rationale as GridLAB-D. |
| **STEPS** | AC-DC hybrid power system analysis tool. Less established in the US community; documentation primarily in Chinese. Worth revisiting if HVDC modeling becomes a requirement. |
